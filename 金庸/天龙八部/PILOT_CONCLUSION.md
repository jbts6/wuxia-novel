## 核心洞察

1. **LLM 是合格的知识架构师**：对已知小说（金庸/古龙等），LLM 能写出准确、连贯、无冲突的结构化知识——关系图、性格定位、one_line、power_rank 都远优于逐章抽取+合并。
2. **LLM 不是合格的引文抄录员**：LLM 无法精确回忆原文的行号和原句（0% 精确引文），但能给出事件锚点（角色+地名+事件关键词），代码可据此在原文定位（89% 定位率）。
3. **职责分离是核心**：让 LLM 决定"引什么"（anchor），让代码决定"在哪"（locate.js 滑窗打分找共现），这是新方案的工程本质。
4. **dialogues 可走"直接读原文"路径**：与其让 LLM 凭记忆写台词（必然幻觉），不如让 agent 直接 grep 原文章节提取代表性对话。

## 流水线对比

| 维度 | 旧（deconstruct + 4 distill） | 新（generate-kb） |
|------|-------------------------------|-------------------|
| 阶段数 | 5+（抽取 + 4 个 distill） | 3（split + generate + locate） |
| LLM 调用次数 | 50 章 × 抽取 + 4 × distill | 3 次 pass（角色/功法/细节） |
| 关系图冲突 | 大量（合并不可避免） | 0（整体生成） |
| one_line 质量 | 随机章节事件 | 全书定位 |
| alias 质量 | 泛称污染 | 真实称呼 |
| 引文精度 | 高（逐章抽取） | 89% 定位（event anchor） |
| 适用范围 | 任意小说 | **仅限训练数据里的知名小说** |
| 失败模式 | 合并冲突、distill 兜底成本高 | 小众小说 LLM 不知道 |

## 建议

1. **对金庸全集、古龙代表作等知名小说**：用 generate-kb 替代 deconstruct + distill，质量更好、速度更快、成本更低。
2. **对网文、小众小说、原创作品**：保留旧 deconstruct + distill 流水线。
3. **对 generate-kb 的已知问题**：
   - dialogues 62.8% grounded 率拖后腿 → 后续可用专门 agent 直接读原文每章提取 10 条代表性台词
   - 40 个 `located_different_chapter`（LLM 猜错章节）→ 扩大 anchor 词典（加更多角色别名/事件词）可改善
   - 52 个 not_located → 同上

## 后续优化点

1. 扩大 `locate.js` 的 anchor 词典：把 mention_index.jsonl 里所有 term 都加入（已做），可再加别名映射（如"段公子"→"段誉"）。
2. 用 `prompts/pass3-patch.md` 跑补丁 pass：针对 not_located 的 52 条 + 错章的 40 条，让 LLM 给出更好的 anchor。
3. dialogues 可单独用 agent 读原文提取：每章挑 10 条代表性台词，避免 LLM 幻觉。
