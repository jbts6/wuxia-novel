# 优化知识库 Dashboard 浏览与更新工作台

## Goal

将现有 Dashboard 改造成固定桌面端的只读知识库工作台，让用户能够稳定浏览已有知识库，并通过清晰的产物状态和下一步建议降低外部更新与校验成本。

## Background

- 产品仅面向固定桌面端，不考虑移动端、响应式断点或不同分辨率适配。
- 当前 Dashboard 使用 React、Vite、React Router、Zustand 和本地知识库 JSON。
- `dashboard/public/data/books.json` 已包含 83 本书的元数据，但运行时仍从 `dashboard/src/data/books.ts` 静态导入少量书籍，存在两套书目来源。
- `dashboard/src/data/books.ts:8` 静态导入尚无 `data/` 目录的《鸳鸯刀》，当前生产构建报 `TS2307`。
- 不同书籍的数据 Schema 存在漂移。以《天龙八部》为例，部分实体只有 `id`、`name`、`source_refs`，而详情页在 `dashboard/src/pages/Characters.tsx:154` 直接读取 `personality.traits`，点击人物会导致白屏。
- 现有列表按书内实体分类，缺少跨书籍、跨实体类型的统一检索入口。
- 现有 `dashboard/scripts/sync-books.ts` 通过修改 alias、tsconfig 和 `books.ts` 接入书籍，更新链路不是单一数据源，也没有发布前 Schema 门禁。
- 《射雕英雄传》的对话数据超过 9000 条，`dashboard/src/pages/Dialogues.tsx:95` 当前会直接渲染全部匹配记录。
- 仓库已有 `docs/superpowers/specs/2026-06-19-global-library-dashboard-design.md`，其中关于动态加载、来源元数据、缺失文件容错、筛选和导出的方向应优先复用。
- 当前四个作者目录中共有 91 本具有原文 `.txt` 的书，可作为管理总览的基础书目。
- 新版流水线以 `build/scan-manifest.json` 的 `required_window_ids` 和三类 pass 的 `completed_window_ids` 计算扫描进度；当前《陆小凤传奇》三类扫描均为 450/450，《天龙八部》的 named-inventory 和 event-dialogue 为 224/224，gap-audit 为 0/224。
- 当前只有少量书籍具有新版 manifest；部分旧知识库有完整 JSON 和旧版质量报告，但报告没有 `completion_gate_passed` 或 G1-G5，不能等同于新版校验失败。
- `generate-kb` 的正式完成条件是八类 `data/*.json` 均存在，且 `reports/quality_report.json` 中 `completion_gate_passed: true`、G1-G5 全部通过。

## Requirements

### Delivery Scope

- 第一阶段交付知识管理工作台与数据基础：全库状态总览、只读扫描、手动刷新、书籍详情面板、建议动作与复制命令、单一书目数据源，以及当前构建失败和人物详情白屏修复。
- 第一阶段保留现有知识浏览入口，确保已有可浏览书籍能够继续进入原有浏览页面。
- 第二阶段交付知识浏览体验重构：跨书统一搜索、大列表分页或虚拟化、实体列表与详情重构，以及搜索状态恢复。
- 两个阶段分别验证；第一阶段不得依赖第二阶段完成后才能投入使用。

### R1. 固定桌面端信息架构

- 使用固定桌面工作台布局，不实现移动端或响应式变体。
- Dashboard 首屏为“全库生成状态总览”，集中呈现未生成、生成中、可浏览和已完成等全库状态；“全库搜索”在第二阶段作为一级导航页启用。
- 全库总览使用“顶部状态统计 + 下方紧凑书籍表格”的主体结构；状态统计可用于快速过滤表格，表格支持排序与组合筛选。
- 顶层明确区分“知识浏览”和“知识管理”两个工作模式。
- 书籍、实体分类、列表和详情之间应保持稳定、可预测的导航关系。
- 点击任意书籍表格行统一打开右侧详情面板，不因行点击直接离开总览；只有明确操作才进入知识浏览页。

### R2. 单一书目与按需加载

- 书目必须来自一次只读扫描生成的统一目录响应，不再手工维护多套静态 import、alias 或书目文件。
- 工作台书目以仓库内作者和书籍目录为事实来源；存在原文但尚无知识库产物的书籍也必须显示。
- 书籍数据按需加载；单本书加载失败不得阻断其他书籍和全库入口。
- 未满足可浏览数据契约的书籍不得进入知识浏览书目，但仍必须保留在管理总览中。

### R3. 数据契约与容错

- 为 Dashboard 消费的数据定义明确的 Schema 版本和标准化层。
- 兼容数据通过 adapter 归一化；缺失字段显示明确的缺失态，不得触发整页崩溃。
- 页面和书籍级别提供错误隔离、失败摘要和恢复入口。

### R4. 全库浏览

- 支持跨作者、书籍和实体类型的统一搜索。
- 支持按作者、书籍、实体类型和实体特有字段进行筛选和排序。
- 搜索结果必须显示来源书籍、实体类型、关键命中内容和可用的原文证据。
- 用户可以从全库结果进入单书视图，并能返回时恢复搜索、筛选和滚动状态。

### R5. 实体列表与详情

- 桌面列表支持紧凑密度、排序、组合筛选、清空筛选、结果计数和可配置列。
- 大列表使用分页或虚拟化，避免一次渲染数千条记录。
- 详情区域展示基础信息、关系、相关实体、登场章节、原文证据和数据状态。
- 表格行和详情操作应支持键盘访问，搜索和筛选控件应具有明确标签。

### R6. 知识管理工作台

- 工作台只读本地产物，不启动 LLM、不执行终端命令、不管理外部进程，也不修改知识库文件。
- 状态在进入页面时扫描一次，并允许用户手动刷新；不监听文件系统、不持续轮询，也不自动刷新。
- 总览显示上次扫描时间，使用户能够判断当前状态数据的新鲜度。
- 每本书显示生成状态、阶段进度、数据完整度、Schema 版本、失败项、最后更新时间和可浏览状态。
- “生成状态”“校验状态”和“是否可浏览”必须分别计算，避免把旧版完整产物误标为生成失败。
- 生成进度从原文、切章、扫描 manifest、partial、合并产物、`data/` 和校验报告等结构化文件推导，不解析终端 stdout。
- 不计算覆盖整个流水线的综合完成百分比；归并、丰富、人工审核和质量门禁不得使用任意权重伪装成精确进度。
- 对可客观计数的扫描阶段，展示当前 pass 的已完成窗口数、总窗口数，以及三类 pass 的总扫描覆盖计数。
- 书籍表格行以当前阶段作为主状态，以扫描覆盖计数作为辅助信息。
- 生成状态至少区分：未生成、准备完成、扫描中、待归并、待查漏、数据已产出。
- 校验状态至少区分：未校验、旧版未证明、校验失败、G1-G5 通过。
- 可浏览状态根据消费接口所需 JSON 是否齐全、可解析且满足 Dashboard 数据契约计算。
- 具有完整、可解析消费 JSON 的旧知识库允许进入知识浏览区；若未通过新版门禁，必须显示“待新版验证”，且不得计入“已完成”数量。
- 只有 `completion_gate_passed: true` 且 G1-G5 全部通过的书籍才能标记为“已完成”。
- 对未完成或失败的书籍，工作台显示缺失产物、失败阶段、建议的下一步动作和可复制的终端命令，但只生成命令文本，绝不负责执行。
- 建议命令必须来自集中维护的阶段映射，并包含明确的目标书籍上下文，避免页面散落硬编码或给出含糊命令。
- 未完成书籍的详情面板展示当前阶段、扫描计数、缺失产物和校验问题；可浏览书籍在相同面板中额外提供明确的“进入知识库”操作。
- 只有满足完整产物条件的书籍才进入知识浏览数据源；其他书籍仍显示在管理总览中。
- 抽取数据与人工标注分离，人工标注不得直接覆盖来源证据。

### R7. 视觉方向

- 保留衬线标题和暖色品牌强调，但内容区使用更中性的桌面数据工作台表面。
- 优先保障信息层级、扫描效率和操作密度，避免大面积空白卡片和单一米黄色背景。
- 使用一致的图标、状态色、间距和交互反馈。

## Out of Scope

- 移动端页面和触控导航。
- 响应式断点和多分辨率布局适配。
- 第一阶段不进行跨书籍同名实体自动去重或合并。
- 为视觉优化修改知识抽取源数据。
- 从浏览器启动 LLM、执行任意终端命令或监听终端 stdout。
- 在第一版中暂停、重试、终止或接管知识库生成进程。

## Acceptance Criteria

### 第一阶段

- [ ] 任意单本书缺失或损坏时，Dashboard 其他可浏览书籍仍可正常进入。
- [ ] 首次进入 Dashboard 时显示全库生成状态总览。
- [ ] 全库总览顶部显示关键状态统计，下方表格可按状态过滤，并可对生成阶段、扫描计数、校验状态和更新时间进行浏览或排序。
- [ ] 点击任意书籍行打开右侧详情面板；未完成书籍不会进入空白浏览页，可浏览书籍只能通过明确按钮进入知识库。
- [ ] 未通过文件完整性、Schema 校验和加载冒烟测试的书籍不会进入知识浏览书目。
- [ ] 点击任意实体详情不会因可选字段缺失导致整页白屏。
- [ ] 管理视图能够显示每本书的生成阶段、校验状态、失败原因、更新时间、可浏览状态和完成状态。
- [ ] 进入管理总览或点击刷新时执行一次只读状态扫描，并更新上次扫描时间；页面空闲期间不会监听文件或自动轮询。
- [ ] 存在原文但尚未生成知识库的书籍也会出现在管理总览中，并明确标记为未生成。
- [ ] 正在生成的书籍可以根据 manifest 和 partial 等产物显示已完成数量、总数量和当前阶段。
- [ ] 工作台不展示全流程综合百分比；扫描阶段只展示 manifest 中可验证的真实窗口计数。
- [ ] 只有八类消费 JSON 齐全、可解析并满足 Dashboard 最低数据契约的书籍会进入知识浏览入口。
- [ ] 具有完整消费 JSON 但未通过新版门禁的书籍可浏览，并始终显示“待新版验证”状态。
- [ ] “可浏览知识库”和“已完成知识库”使用不同统计口径；旧版待验证知识库不会计入已完成数量。
- [ ] 工作台不提供任何执行终端或 LLM 命令的能力。
- [ ] 未完成书籍能够显示与当前阶段匹配的下一步动作和可复制命令；复制操作不会触发任何命令、文件写入或进程启动。
- [ ] 刷新、筛选、表格行、复制命令和详情关闭操作均可通过键盘完成。
- [ ] 第一阶段的 `npm run lint`、`npm test`、`npm run build` 和桌面端 E2E 测试通过。

### 第二阶段

- [ ] 全库搜索作为一级导航页启用。
- [ ] 用户可从一个入口搜索至少人物、武功、物品、势力和地点，并按作者与书籍筛选。
- [ ] 全库结果展示来源信息，并可跳转至对应单书详情。
- [ ] 返回全库列表时保留查询条件、筛选条件和浏览位置。
- [ ] 超过 5000 条记录的列表不会一次性创建全部内容节点。
- [ ] 搜索、筛选、表格行和详情关闭操作均可通过键盘完成。
- [ ] 第二阶段新增单元测试、构建检查和桌面端 E2E 测试通过。
