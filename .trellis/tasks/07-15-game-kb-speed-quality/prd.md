# 优化 generate-game-kb 速度与重点质量

## Goal

融合历史 `deconstruct-novel` 的逐章并发与确定性注册表，以及 `distill-*` 的领域联合清洗思路，缩短 `generate-game-kb` 的语义后处理链路；在不牺牲原文溯源和九类 JSON 兼容性的前提下，让中短篇小说能够在可接受时间内产出游戏知识库。

## Requirements

- 保留现有 fresh v2 安全合同：每章由一个隔离 Worker 完整读取，一个 Worker 只处理一章；Worker 只写草稿，主线程串行 `accept`，不得复用旧 `data/` 或旧 run 作为生成输入。
- 并发槽总数保持 4：主线程之外最多 3 个 Worker。逐章提取和互不依赖的领域后处理都应使用固定 3 Worker 池。
- 将逐类别、逐 shard 的重复 `merge` / `clean` 收敛为一次确定性预处理和四个联合语义域：
  - 剧情域：角色、事件、关键对白。
  - 武学域：武功、招式。
  - 物品域：只保留秘籍、剧情关键物、高级药毒、神兵利器及其他稀有特殊物品。
  - 世界域：势力、地点，采用较低完整度标准。
- 角色、武功、招式、关键物品、事件是高优先级类别；地点、势力、普通对白等不得拖慢全书收尾，其非结构性质量问题只产生告警。
- 领域语义单元必须同时完成别名归并、噪音删除和证据合并。正常路径每个领域只提交一次；每个领域最多允许一次格式修正和一次语义补救，禁止重新展开成旧式全类别循环。
- 语义处理前先机械完成名称归一、精确去重、候选注册、引用迁移和证据合并；模糊项保留为待判定候选，禁止为通过校验而误删。
- `chapter_summaries` 从已接受的事件和人物投影中机械生成，不新增 AI 单元。
- 补漏只允许针对高优先级类别，并由确定性覆盖检查触发；不得对低优先级类别做全书重扫。
- 输出仍安装为现有九类兼容 JSON，保持稳定 ID、引用闭包、证据章号、归档和可恢复性合同。
- 新增分阶段计时、AI 单元数、尝试次数、最大输入字节和补救原因统计，能够直接定位速度回退。
- 不修改 `.agents/skills/generate-kb/`；本任务只改造 `.agents/skills/generate-game-kb/` 及必要的 Trellis 规范/证据。

## Acceptance Criteria

- [ ] 自动化测试证明确定性候选注册表可完成名称归一、精确去重、证据合并和引用迁移，且不会静默删除模糊候选。
- [ ] 自动化测试证明九类章节候选被稳定路由到四个语义域，领域结果可确定性组装回九类兼容 JSON。
- [ ] 正常路径不再创建逐类别 `merge:*:<shard>` 与 `clean:*:<shard>` AI 工作量；四个领域各最多一个初始语义单元，`chapter_summaries` 的 AI 尝试数为 0。
- [ ] 高优先级质量门能阻断未解析引用、缺失证据、武功/招式混淆、关键物品噪音和重要事件缺失；世界域及普通对白的非结构性缺口只告警。
- [ ] 定向 recall 只在高优先级覆盖门失败时产生，并且有界、可审计，不触发全书 merge/clean 重跑。
- [ ] 相关单元、集成和 Skill 合同测试通过；已知 Windows archive/symlink 权限类基线失败须如实记录，不得误报全绿。
- [ ] 对 `金庸/雪山飞狐` 执行 fresh v2 冷启动：归档旧数据、不复用旧产物，完整生成、验证、安装并归档 run。
- [ ] 《雪山飞狐》端到端目标耗时不超过 45 分钟，硬上限 60 分钟；若超过硬上限，本任务不得宣称速度验收通过。
- [ ] 《雪山飞狐》的角色、武功、招式、关键物品、事件均通过重点质量门和原文证据抽样，安装哈希与归档哈希可复核。

## Notes

- 旧流程历史分析表明，主要耗时来自重复语义后处理，而不是逐章完整读取：此前《飞狐外传》产生 25 个 merge 与 15 个 clean，《笑傲江湖》产生 38 个 merge 与 19 个 clean。
- 《雪山飞狐》约 392 KB、10 章，适合作为中短篇冷启动速度样本。
