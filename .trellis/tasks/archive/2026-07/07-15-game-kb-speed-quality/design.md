# generate-game-kb 速度与重点质量平衡设计

## 1. 设计目标

当前 v2 流程已经具备可靠的逐章隔离、串行验收、确定性组装和可恢复性，但逐类别分片后的 merge/clean 仍会放大 AI 工作量。新设计保留这些可靠性边界，把后处理从“很多类别反复判断”改成“机械预清洗 + 四域一次语义决策”，将模型时间集中在用户最看重的角色、武功、招式、关键物品和事件。

《雪山飞狐》的验收目标是 cold fresh v2 端到端不超过 45 分钟，60 分钟为硬上限。速度达标不能以旧产物复用、跳过完整章节读取或降低原文证据要求换取。

## 2. 不变合同

- 每个章节单元只绑定一个章节文件，由隔离 Worker 完整读取。
- 章节 Worker 不读取整书聚合产物，不执行 `accept`，不修改 `progress.json`。
- 主线程逐个、串行执行 accept，所有提交继续受输入哈希和尝试预算约束。
- 同一时刻主线程之外最多 3 个 Worker。
- 新书验证必须创建 fresh v2 run；旧 `data/`、旧 run、百科、影视改编和模型记忆都不是生成输入。
- 最终仍生成九类 JSON，保持稳定 ID、引用闭包、安装验证和不可变归档。
- `manual_review`、震荡检测和不得自动 `reset-unit` 的边界不变。

## 3. 新数据流

```text
10 个章节原文
  -> 3 Worker 并发逐章提取
  -> 主线程串行 accept
  -> 确定性候选注册表
       名称规范化 / 精确去重 / 证据合并 / 引用迁移
  -> 四个独立语义域（最多 3 Worker 并发）
       剧情域 / 武学域 / 物品域 / 世界域
  -> 主线程串行 accept
  -> 确定性九类组装 + chapter_summaries 投影
  -> 高优先级覆盖检查
       仅失败类别产生定向 recall
  -> 分层质量门
  -> build-final / verify / install / archive
```

常规路径的 AI 工作量为 10 个章节单元、4 个领域单元和 1 个重点抽样单元。只有确定性高优先级覆盖门失败时才增加定向 recall；不存在 `merge:book`、`clean:book` 或逐类别 clean 重跑。

## 4. 确定性候选注册表

注册表以类别与规范化名称为主索引，以章节候选的稳定临时 ID 为引用入口。它只做高置信机械操作：

1. 统一首尾空白、兼容标点、可证明的类别前后缀和完全一致别名。
2. 合并完全相同的类别/规范化名称候选，保留所有来源章节和证据片段。
3. 建立旧候选 ID 到注册项 ID 的迁移表，并重写人物、事件、招式、物品等交叉引用。
4. 把同名异类、近似名、身份未决和证据冲突标为 `pending`，交给对应领域单元判断。

注册表不得使用模糊相似度直接删除候选。这样继承 `merge-entities.js` 的速度优势，同时保留 `distill-*` 对模糊数据谨慎处理的质量边界。

## 5. 四个联合语义域

### 5.1 剧情域

联合处理角色、事件和关键对白。角色身份合并必须同步迁移事件参与者和对白说话人；事件去重必须保留跨章证据。只保留能够服务关键冲突、奇遇、传承、反转或关系转折的对白，每个最终事件最多一条。角色分级仍为核心、重要、次要、龙套、背景，只有前两级要求丰富描述。

### 5.2 武学域

联合处理武功与招式，以解决名称层级和从属关系。原文明确定名的功法与招式优先高召回；“拍出一掌”“全力一挥”等普通动作必须删除。招式记录必须指向可解析的武功，无法确认从属时保留证据并进入人工/定向补救，不得猜测。

### 5.3 物品域

只保留秘籍、剧情关键物、高级药毒、神兵利器及稀有特殊物品。衣食器具、普通兵器和一次性布景默认机械标记为低价值，再由单个领域单元处理边界项。关键物品必须能链接相关人物或事件并具有原文证据。

### 5.4 世界域

联合处理势力与地点，只要求身份不混淆、核心引用可解析和基本证据存在。地理层级、次要场景完备度及边缘势力召回不足记为 warning，不阻塞安装，也不触发全书补漏。

每个领域初始只有一个 AI 单元。格式不合法时允许一次格式修正；语义质量门明确指出缺陷时允许一次语义补救。两者都必须沿用相同输入哈希和现有总尝试预算，不能拆分成新的类别循环。

## 6. 确定性九类组装

领域 accept 后输出的是“保留/合并/删除决策 + 规范记录 + 引用迁移”。组装器按固定类别映射写回 characters、events、dialogues、skills、techniques、items、factions、locations，并从事件记录机械投影 chapter_summaries。投影只摘要该章已接受事件、关键参与者和结果，因此不调用模型。

组装阶段必须验证：

- 每个迁移前 ID 恰好解析到一个最终 ID。
- 高优先级记录至少有一条合法章节证据。
- 事件参与者、招式所属武功、关键物品关联对象均可解析。
- 删除决策有原因码，pending 决策不会被静默丢弃。

## 7. 分层覆盖与质量门

### 硬门（阻塞 build/install）

- JSON Schema、稳定 ID、引用闭包、证据章号与归档完整性。
- 核心/重要角色、武功、明确招式、关键物品和关键事件的候选闭包。
- 武功/招式类型与从属关系正确，普通动作不得混入招式。
- 关键物品具备稀有性或剧情作用，关键事件具备参与者、起因/结果和证据。
- 重点抽样的原文支持率达到既有 95% 合同；低于阈值直接人工复核，不回滚整书流程。

### 软门（warning）

- 次要地点、边缘势力、普通对白和后三档人物的描述深度与完整度。
- chapter_summaries 的文采与非关键细节覆盖。

确定性覆盖检查只对失败的高优先级类别生成 `recall:<category>`。Recall 输入包含缺口清单和相关章节证据窗口，不重新提交完整领域，也不为地点、势力或普通对白启动 recall。

## 8. 计时与可观测性

run 元数据记录：

- prepare、chapter extraction、registry、domain distill、recall、quality、install、archive 各阶段起止时间和墙钟耗时。
- AI 单元按类型的计划数、完成数、总尝试数、格式修正数、语义补救数。
- 每个 AI 单元的输入字节数及全 run 最大值。
- 候选数量在章节输出、机械注册、领域决策、最终组装各阶段的变化。
- hard failure、warning 和定向 recall 的原因码。

这些指标写入 run 内的机器可读 JSON；测试和《雪山飞狐》验收报告直接消费该文件，避免人工从日志估算。

## 9. 兼容、回滚与失败边界

- 新流程使用新的语义合同版本或显式 fast-domain profile 标识。旧的未完成 run 不原地升级；必须归档后 fresh prepare。
- 保留现有最终九类 Schema 和安装目录合同，因此消费者无需迁移。
- 若四域组装发现无法唯一解析的引用，进入 `manual_review`，而不是回退旧式 merge/clean。
- 代码回滚只需移除新 profile/合同路由；既有归档 run 仍可解压审计。
- 《雪山飞狐》超过 60 分钟、重点质量门失败、或需要复用旧数据才能通过时，速度验证判定失败并保留完整证据。

## 10. 验证策略

先以 TDD 增加注册表、领域路由、确定性组装、分层质量门、定向 recall 和计时统计测试，再修改实现。随后运行聚焦单测、集成测试、Skill 合同测试和可运行的全套测试。最后归档《雪山飞狐》旧数据，执行 10 章 fresh v2 冷启动，核验九类安装结果、重点类别原文抽样、运行/安装/归档哈希与阶段耗时。
