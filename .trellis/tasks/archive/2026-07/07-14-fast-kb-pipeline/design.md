# generate-game-kb 技术设计

## 1. 设计目标

新增独立 Skill `.agents/skills/generate-game-kb/`，用简单的分章文件流程生成面向武侠游戏素材准备的知识库。现有 `.agents/skills/generate-kb/` 保持不变，新流程不读取或写入其 managed run、事件日志、锁、`.kb/current` 或 bundle。

新流程的核心目标：

- AI 直接逐章理解原文，优先保证约 95% 的实用内容质量。
- token 消耗不作为指标；中篇《飞狐外传》目标 60 分钟内，约 50 章长篇目标 90 分钟内。
- 产出与当前消费者兼容的九类 JSON，并从只含原著、`ch_split/` 和 `_archive/` 的干净输入基线启动。
- 每次运行按 `run-id` 完全隔离，结束后把 source 快照、草稿、接受产物、报告、日志、进度和 final 数据整体归档。
- 功法、招式、经典桥段、鲜明人物和标志性物品偏向未来武侠游戏素材价值。
- 用候选覆盖账本和条件式类别补漏防止重要物品、对白等高价值内容无解释消失，同时保持正常作品只执行一遍主流程。
- 任何自动纠错都有持久化预算和无进展熔断，不能靠重启刷新，也不能在 ID 之间反复震荡。

## 2. 非目标

- 不修复、重构或复用现有 `generate-kb` 的受管状态机和发布机制。
- 不追求 100% 长尾召回、精确行号、精确段落或逐字段证据门禁。
- 不为达到数量区间而补造实体。
- 不收集普通日用品、普通动作、通用攻击、移动姿态或战斗动画动作。
- 不使用百科、影视改编或模型先验作为事实来源。
- 不自动发布到 `.kb/current`，也不引入版本指针或事务事件日志。

## 3. 方案选择

### 3.1 已选方案：run 隔离的分章文件型 + 条件式补漏

主模型为每章派发一个隔离的原生子代理，每个子代理直接完整读取一个章节并生成独立提取文件。输入哈希未变化且已完成的章节直接跳过。全部章节完成后，先执行确定性覆盖检查；无异常时执行一次全书合并、一次清理和一次最终抽样，存在候选归零、对白覆盖不足、抽样缺类或引用断裂时，只为受影响类别执行一次有界补漏。

所有中间文件位于 `.game-kb-work/runs/<run-id>/`。新 run 之前先把上一轮构建产物整体归档，避免共享工作目录、旧 `data/` 或旧报告参与新运行。主模型仍是唯一状态执行者，只有互不依赖的逐章阅读使用受限原生子代理并行；该方案在速度、可恢复性、隔离和一致性之间最均衡。

### 3.2 未选方案

- 无持久化上限的并行型：初始更快，但遇到宿主限流时会重复失败，且恢复后无法保持退避状态。
- 全书单次型：步骤少，但容易遗漏、截断，失败时几乎必须全部重来。
- 固定“骨架 + 完整”两遍型：召回更稳定，但让每本书都承担第二次全书读取；本设计只在确定性异常时执行类别级补漏。
- 共享 `.game-kb-work/` 后置快照型：改动较小，但运行中仍可能读取旧草稿或报告，不能证明跨 run 隔离。


`schemas.md` 是新流程的独立九类数据契约。它保持当前消费者需要的文件名和字段形状，但不继承现有 `generate-kb` 的逐字段证据门禁。

### 4.2 小说工作目录

```text
<小说目录>/
├── <小说名>.txt
├── ch_split/
├── data/                              # 当前已安装九类数据
├── reports/                           # 当前消费者报告
├── .game-kb-work/
│   └── runs/<run-id>/
│       ├── run.json
│       ├── manifest.json
│       ├── artifact-manifest.json
│       ├── worker-pool.json
│       ├── progress.json
│       ├── source/chapters/ch_NNN.txt
│       ├── drafts/
│       ├── accepted/
│       │   ├── chapters/ch_NNN.json
│       │   ├── recalls/<category>.json
│       │   ├── merged/book.json
│       │   ├── supplements/<category>.json
│       │   └── cleaned/book.json
│       ├── materialized/
│       │   ├── candidates.json
│       │   └── merged-with-supplements.json
│       ├── reports/
│       │   ├── coverage.json
│       │   ├── candidate-resolution.json
│       │   └── run-summary.json
│       ├── final/data/*.json
│       ├── final/reports/*.json
│       └── manual_review.json
└── _archive/
    ├── <timestamp>-before-<run-id>/
    │   ├── archive-manifest.json
    │   └── <上一轮构建的原相对路径>
    └── generate-game-kb/<run-id>/      # 完整 run 归档
```

- `run.json`：`run_id`、原著哈希、状态、开始/安装/归档时间、各阶段与总耗时。
- `manifest.json`：原文路径、全书哈希、章节顺序、章节标题和每章哈希。
- `artifact-manifest.json`：本次 run 生成的全部其他文件、相对路径、类型、大小和哈希；完整归档以它为准，归档回执另行记录 manifest 自身哈希，避免自引用。
- `worker-pool.json`：章节子代理的初始/当前并发上限、429 批次历史与停止状态；恢复时保留，新 run 才重置。
- `progress.json`：每个 AI 单元的输入哈希、提交次数、输出哈希、错误指纹和状态。
- `drafts/`：每次 AI 提交的不可变草稿，便于诊断，不作为最终事实源。
- `accepted/`：通过结构校验的章节、补漏、合并、补充和清理结果；接受后纳入哈希链并不可变。
- `materialized/`：由接受产物确定性投影出的下游输入；补漏只能生成新投影，不能原地改写 `accepted/`。
- `coverage.json`：逐章候选覆盖、重要事件对白覆盖和类别异常。
- `candidate-resolution.json`：每个候选在合并与清理后的唯一去向。
- `manual_review.json`：已熔断或无法唯一解析的问题汇总。

所有进度文件和已接受产物都通过临时文件加同目录重命名写入。CLI 不使用可漂移的当前发布指针；如果存在多个未归档 run，调用方必须显式提供 `--run <run-id>`。

## 5. 命令边界

单一入口 `scripts/flow.js` 只提供有限命令：

- `archive-existing <novel>`：先列出归档计划，再把除唯一原著、`ch_split/`、`_archive/` 外的全部根级条目移动到新归档并验证允许列表。
- `prepare <novel> [--run <run-id>]`：从干净基线发现原文、校验或重建切章、固定 source 快照并创建或恢复 run。
- `status <novel> [--run <run-id>] [--json]`：只报告完成、缺失、失效、覆盖缺口和人工处理项，不返回命令型 `next_action`。
- `worker-backoff <novel> --run <run-id> --batch <batch-id> --reason 429`：同一章节派发批次只记录一次 429，并持久化折半并发；它不调用模型、不提交草稿，也不修改 `progress.json`。
- `check-coverage <novel> --run <run-id>`：确定性生成逐章候选和重要事件对白覆盖报告，必要时开放 `recall:<category>` 单元。
- `accept <novel> --run <run-id> --unit <unit> --draft <path>`：记录章节、补漏、合并、补充、清理或质量抽样提交，校验后接受或记录错误。
- `check-resolution <novel> --run <run-id>`：确定性验证候选去向，必要时开放 `supplement:<category>` 单元。
- `build-final <novel> --run <run-id>`：由 cleaned 结果生成稳定 ID、重写引用并构建九类 final JSON。
- `verify <novel> --run <run-id> [--installed]`：默认检查指定 run 的 final；`--installed` 检查已安装的 `data/`、命名报告和安装回执。
- `install <novel> --run <run-id>`：在所有门槛通过后一次性安装 `data/` 和消费者报告。
- `archive-run <novel> --run <run-id>`：仅在已安装结果复验成功后，把完整 run 移入 `_archive/generate-game-kb/<run-id>/`。
- `reset-unit <novel> --run <run-id> --unit <unit> --confirm`：仅供用户明确授权后重置指定人工处理项。

这些命令不执行模型调用。Skill 指令负责让主模型编排章节子代理或亲自处理后续语义阶段、生成草稿，并由主模型调用 `accept`。CLI 没有 `advance`、claim、lease、stage lock、事件日志或自动动作循环。

## 6. 三个主要 AI 阶段与条件分支

### 6.1 阶段一：逐章阅读与提取

主模型从 manifest 与 `progress.json` 选择待处理章节，为每章分配一个隔离的宿主原生子代理。`prepare` 在新建和恢复 run 时都保证 `<run-dir>/staging/` 存在；每个子代理只读取一个完整章节，并只写主模型指定的 `<run-dir>/staging/chapter_NNN_attempt_<NN>.json`。路径同时绑定 run、unit 和持久化 attempt，`/tmp/chapter_NNN_draft.json` 之类的任意临时路径不属于流程合同。

子代理只读取 `schemas.md`、逐章提示词、本章 source 文件和 manifest 条目，不调用 `accept`，不写 `progress.json` 或 accepted 产物，并只向主模型返回草稿路径和状态。主模型可以按宿主槽位并行派发互不依赖的章节，但必须串行执行 `accept`。CLI 从当前输入哈希和 `progress.json` 计算唯一允许的下一个 attempt，即持久化 `attempts + 1`；`accept` 只接收当前 run 中对应的精确 staging 路径。提交记账后，CLI 将原始内容复制进 `drafts/`，并在成功或拒绝两种结果下都消费 staging 文件；子代理上下文随即丢弃。

新 run 的持久化章节并发上限为 10。每批实际派发数为当前上限、宿主可用槽位和待处理章节数的最小值。同一批一个或多个明确 429 只执行一次 `worker-backoff`，上限按 `10 → 5 → 2 → 1` 变化；429 不进入章节提交预算，重试使用新子代理完整重读原文。当前 run 不自动升并发，恢复和上下文压缩保留上限；并发为 1 时再次发生新 429，状态转为停止并报告外部限流。

恢复或上下文压缩后，主模型重新读取 Skill、schema、阶段提示词和一次 `status --json`，只依赖落盘状态继续。已 `done` 的章节不重读；非 `manual_review` 单元只接受 `attempts + 1` 的 staging 文件。编号较小的残留文件由 CLI 以 `DRAFT_STAGING_MISMATCH` 拒绝且不消耗预算；下一个 attempt 尚无文件时，由新的子代理从头完整读取。每个章节文件包含：

```json
{
  "chapter": 3,
  "title": "第三章 ...",
  "source_hash": "sha256:...",
  "characters": [],
  "events": [],
  "items": [],
  "skills": [],
  "techniques": [],
  "factions": [],
  "locations": [],
  "dialogues": [],
  "summary": {},
  "coverage": {}
}
```

章节候选使用原著名称和章节内临时键，不生成最终 ID。跨类关联暂时使用名称或本章临时键。

每条候选至少包含：

- 原著名称、类别、重要性和关键事实；
- 当前章节号；
- 一个短原文锚点 `text`；
- 必要的关联名称。

对白提取与最终筛选分离。核心/重要事件记录 `importance` 和 `quote_status`；只要原文存在适合引用的直接对白，逐章阶段可以为同一事件保留多条短候选。最终“每事件最多一条”不能成为逐章候选上限。`coverage` 至少记录各类候选数、重要事件数、可引用事件数和对白候选数。

### 6.2 阶段二：全书合并与去重

脚本先按类别和规范化名称归组完全相同的候选。AI 只处理真正有歧义的别名、同名异人、跨章事件和实体边界。

合并规则：

- 同一实体的章节引用去重后合并。
- 跨章事件保留多个不连续章节，不压缩成单章，也不要求列出每个过渡章节。
- 同名异人不能仅凭名称合并；需要角色、关系和章节上下文支持。
- 同一事件中的对白在合并后引用事件临时键。
- 每个逐章候选必须在覆盖账本中记录 `merged_to`、`rejected` 或 `ambiguous`；`rejected` 必须有可验证原因。
- 无法唯一决断的合并项进入人工清单，不反复尝试 ID 方案。

### 6.3 阶段三：一次清理

清理只执行一次全书语义判断，处理：

- 武器、物品、功法和招式的类别纠正；
- 描述词、泛称和普通动作伪实体删除；
- 普通物品删除与重要物品保留；
- 五级人物的详细/简略投影；
- 事件附属对白精简；
- 游戏素材索引候选标记；
- 为被删除的高价值候选写出结构化原因。

清理不因数量区间补数或删数。接受前必须验证高价值候选去向和上游哈希；完成后不得再次执行全量清理。

### 6.4 条件式类别补漏

确定性检查只在以下情况开放补漏单元：

| 异常 | 条件 | 单元 |
|---|---|---|
| 候选归零 | 已有候选组，合并或清理后为 0，且去向/拒绝理由不完整 | `supplement:<category>` |
| 物品空库 | 正文不少于 15 万汉字或核心/重要事件不少于 20 个，且物品候选为 0 | `recall:items` |
| 对白覆盖不足 | 可引用核心/重要事件的最终对白覆盖低于 70% | `recall:dialogues` |
| 章节分布异常 | 可引用核心/重要事件分布在至少 8 章，而对白候选覆盖的这些章节少于 30% | `recall:dialogues` |
| 引用断裂 | 候选或实体引用无法唯一解析 | `supplement:<category>` 或人工处理 |

`recall:*` 只读取缺口报告列出的章节和相关已接受事件；`supplement:*` 只读取候选账本和当前合并结果。每类最多一次语义补漏，另允许一次纯格式修正。确定性脚本把接受的补漏包投影成新的 `materialized/candidates.json` 或 `materialized/merged-with-supplements.json`；它不改写 `accepted/`，不重新执行全书 AI 合并，也不允许第二次全量清理。合法空库必须留下复核章节范围和 `none_found` 结论。

## 7. 九类最终数据

最终 `data/` 包含：

1. `characters.json`
2. `events.json`
3. `items.json`
4. `skills.json`
5. `techniques.json`
6. `factions.json`
7. `locations.json`
8. `dialogues.json`
9. `chapter_summaries.json`

所有文件顶层为数组。

### 7.1 SourceRef

```json
{
  "chapter": 3,
  "text": "简短原文锚点",
  "line_start": 42,
  "line_end": 45
}
```

- `chapter` 和 `text` 必填。
- `line_start`、`line_end` 可选且只作提示。
- 验收只要求章节正确，不要求行段精确。
- Dashboard 当前已支持没有行号的 source ref。

### 7.2 人物

沿用核心、重要、次要、龙套、背景五级：

- 核心、重要：完整生平、详细性格、关系变化、功法关联和多章依据。
- 次要、龙套、背景：只保留姓名/别名、身份、等级、简短定位、代表性行为或彩蛋价值、必要关系/功法引用和章节依据。
- 简略人物仍输出兼容字段，无信息字段使用空字符串、空数组或空对象，不生成模板化长传记或凑足固定性格词。

### 7.3 功法与招式

- 原文明确定名、可与普通动作区分的功法和招式原则上全部保留。
- 冷门、低频或威力弱只影响重要性，不作为删除理由。
- 功法重点保留效果、限制、修炼条件、克制、阶段、使用者和代表表现。
- 招式必须有正式名称；“全力一挥”“拍出一掌”“纵身跃起”等普通动作直接丢弃。
- 能说明已命名功法机制的动作特征只能写入 `combat_style`、`effects` 或对应招式描述，不能独立成条目。
- 不新增第十类“动作描写”。

### 7.4 事件

优先保留经典冲突、反转、奇遇、传承、复仇、门派斗争和人物关系转折。记录起因、关键过程、结果、参与者、地点、重要性和跨章引用。

### 7.5 物品

只保留：

- 神功秘籍；
- 推动剧情的重要物品；
- 高级或具有特殊作用的毒药、丹药和疗伤药；
- 神兵利器及其他稀有特殊物品。

普通随身武器、日用品和没有特殊能力或剧情作用的物品不收录，不能为满足数量下限而降低准入标准。

### 7.6 对白

- 对白是事件附属资料，每条必须引用有效 `event_id` 和正确章节。
- 最终每个事件最多一条；逐章阶段可保留多条候选。
- 只保留能代表重要事件、关系转折或经典人物形象的短对白。
- 可引用的核心/重要事件必须有最终对白或明确的省略理由，覆盖率低于 70% 时不得进入最终构建。
- 不保留冗长上下文，不要求精确行段。

## 8. 游戏素材索引

额外生成 `reports/game_materials.json`，但不增加第十类事实数据。索引条目只引用九类数据中的稳定 ID，并可包含：

- `material_type`：战斗系统原型、经典剧情桥段、角色原型/彩蛋、标志性物品、门派与世界观素材；
- `source_id`；
- `relevance`；
- `suggested_use`；
- `reason`。

索引不得创造原著之外的实体或设定，也不得反向修改九类事实数据。

## 9. 数量评估

数量按正文汉字数分档：

| 类别 | 短篇 `<15万字` | 中篇 `15–50万字` | 长篇 `>50万字` |
|---|---:|---:|---:|
| 人物 | 10–35 | 25–75 | 45–120 |
| 事件 | 20–80 | 60–220 | 150–400 |
| 重要物品 | 3–12 | 6–20 | 10–30 |
| 功法 | 5–20 | 10–35 | 15–50 |
| 招式 | 0–20 | 5–45 | 10–80 |
| 门派/势力 | 2–12 | 5–25 | 10–40 |
| 地点 | 5–25 | 10–50 | 20–80 |
| 代表性对白 | 5–25 | 20–60 | 40–100 |
| 章节摘要 | 每章 1 条 | 每章 1 条 | 每章 1 条 |

数量检查只执行一次并只生成报告。最终报告可以重新统计实际数量，但数量区间本身不触发 AI 动作，也不阻断写入。只有候选去向、类别覆盖、章节分布或引用完整性出现确定性异常时，才触发第 6.4 节的类别补漏。

## 10. 稳定 ID 与引用重写

AI 从不生成或修补最终 ID。`build-final`：

1. 对清理后的规范名称做确定性规范化和拼音 slug。
2. 按类别增加固定前缀。
3. 对重名使用稳定、可复现且只含 `a`–`p` 字母的短摘要后缀消歧，保持现有小写拼音 ID 正则兼容。
4. 建立“类别 + 规范名称/别名”到最终 ID 的映射。
5. 一次性重写九类 JSON 和游戏素材索引的全部引用。
6. 任何零匹配或多匹配引用进入 `manual_review`，不调用 AI 改 ID。

相同输入重复构建必须产生相同 ID 和相同引用结果。

## 11. 纠错预算与防循环

每个 AI 单元持久化：

```json
{
  "unit": "chapter:003",
  "input_hash": "sha256:...",
  "status": "pending",
  "attempts": 1,
  "output_hashes": ["sha256:..."],
  "error_fingerprints": ["sha256:..."]
}
```

状态只使用 `pending`、`done`、`manual_review`、`stale`，用于文件清单，不构成自动阶段状态机。

规则：

- 每个单元总共最多提交 3 次，包括首次生成和最多 2 次纠错。
- 相同输出连续出现时立即熔断。
- 相同标准化错误连续出现时立即熔断。
- 输出或错误出现 `A → B → A` 震荡时立即熔断。
- 同一输入哈希下重启不重置预算。
- 章节原文哈希变化后，该章成为新输入并获得新预算。
- 熔断后继续处理其他独立章节，但在人工问题解决前禁止写入 `data/`。
- 只有用户明确执行带 `--confirm` 的指定单元重置，才允许开启新预算。
- `recall:*` 和 `supplement:*` 各自最多一次语义生成；剩余尝试只允许修正结构或格式错误。
- 每个接受文件将相对路径和哈希加入上游哈希链；后续发现文件内容与已接受哈希不同，立即报 `ACCEPTED_ARTIFACT_MUTATED`。

错误指纹由排序后的错误码、字段路径和目标引用组成，不包含时间戳、临时路径或自由文本噪声。

## 12. 最终质量检查

### 12.1 必须通过

- 九类 JSON 可解析，顶层为数组，必填字段类型正确。
- ID 唯一、前缀正确，跨类引用全部可解析。
- 引用章节存在；行段精度不参与通过判断。
- 每章恰好一个章节摘要。
- 每条对白引用有效事件。
- 每个逐章候选都有唯一合并/清理去向。
- 可引用核心/重要事件的对白覆盖率不低于 70%。
- 游戏素材索引只引用有效 ID。
- `manual_review.json` 为空。
- 最终内容抽样达到 95%。

### 12.2 95% 抽样

标准样本 40 条：

- 功法/招式 12；
- 事件 8；
- 人物 5；
- 物品 5；
- 对白 4；
- 门派/地点 4；
- 章节摘要 2。

样本由脚本按固定种子、类别配额和稳定 ID 排序确定，AI 不能自行选择样本。类别配额不能转给其他类别；某类记录不足时检查全部记录，合法为空时检查持久化的空库复核结论。每条检查名称、类别、关键事实和引用章节。标准样本至少 38 条通过；不足 40 个可用检查对象的小作品检查全部对象，通过数必须达到 `ceil(样本数 × 0.95)`。抽样低于 95% 直接进入 `manual_review`，不自动触发全库返工。

### 12.3 只报告

- 数量是否超出建议区间；
- 行号、段落和原文锚点是否精确；
- 少量长尾人物、普通物品或次要地点是否遗漏。

## 13. 旧产物归档、安全安装与 run 归档

### 13.1 新 run 前归档

`archive-existing` 先识别唯一原著文本，并生成包含其他所有根级条目的移动计划。保护允许列表只有原著 `.txt`、`ch_split/` 和 `_archive/`。执行时保留原相对路径并写 `archive-manifest.json`；路径冲突、越界符号链接或任一步失败时回滚全部已移动条目。完成后重新扫描根目录，任何额外条目都使操作失败。

`ch_split/` 可以保留，但 `prepare` 必须先在内存中从原著确定性切分，并逐一比较章节数、标题和内容哈希；全部一致时才复制到 run source，否则直接使用从原著得到的切分结果。旧 `build/`、`data/`、`reports/`、`prompts/`、`review/`、`summary.md`、`split-config.json` 和 `.game-kb-work/` 都随上一轮产物归档，不能成为新 run 输入。

### 13.2 安装

`install` 在小说目录生成完整的 `data.next-generate-game-kb-*` 和消费者报告临时目录，复验结构、ID、引用、覆盖账本和 final 哈希后，通过同文件系统重命名安装。因为新 run 已从干净基线启动，安装时不继承未知旧条目；任何意外存在的 `data/`、`reports/` 或活动工作目录都报 `DIRTY_INSTALL_BASELINE`，要求重新归档而不是混写。

### 13.3 完整 run 归档

`verify --installed` 成功后，`archive-run` 校验 `artifact-manifest.json` 并把整个 run 移入 `_archive/generate-game-kb/<run-id>/`。归档回执补写 `archived_at` 和归档哈希，小说根目录只保留原著、`ch_split/`、当前 `data/`、当前消费者 `reports/` 和 `_archive/`，不保留活动 `.game-kb-work`。`run.json` 记录 `started_at`、`installed_at`、`archived_at`、逐章、合并、清理、补漏、脚本和人工等待耗时。

## 14. 验证策略

### 14.1 单元测试

- 章节识别、全书/章节哈希、输入变化和恢复跳过。
- progress 原子写入、损坏 fail-closed 和同输入预算持久化。
- 三次上限、重复输出、重复错误和 `A → B → A` 熔断。
- 名称规范化、别名归组、同名异人和稳定 ID 冲突后缀。
- 全局引用重写、零匹配和多匹配人工处理。
- 章级 source ref 接受、错误章节拒绝、可选行号兼容。
- 五级人物详略投影。
- 普通动作/正式招式、武器/招式、普通物品/重要物品分类。
- 跨章事件合并和每事件最多一条对白。
- 逐章可保留多条对白候选，最终可引用核心/重要事件覆盖率门禁。
- 候选覆盖账本、无解释候选归零、空物品专项复核和类别补漏预算。
- 接受产物哈希链和 `ACCEPTED_ARTIFACT_MUTATED` 防篡改。
- 数量检查只生成报告，不能单独创建补漏任务。
- 游戏素材索引引用完整性。
- 归档允许列表、归档清单、越界链接拒绝、中途失败回滚和 run 隔离。
- 安装、已安装复验、完整 run 归档和运行时间记录。

### 14.2 集成测试

使用小型三章 fixture 验证：archive-existing → prepare → 章节接受 → check-coverage → 条件式补漏 → 合并 → check-resolution → 清理 → build-final → verify → install → verify --installed → archive-run 的完整流程，并注入无效 JSON、悬空引用、ID 碰撞、人工问题、归档故障和安装故障。

另增加三组回归夹具：7 个物品候选在合并后无解释归零、重要事件遍布全书但对白只集中在前部章节、质量抽样因配额重分配而没有物品/对白。三者必须稳定触发对应补漏或阻断。

### 14.3 端到端试跑

真实书籍验证使用 `金庸/笑傲江湖` 和 `金庸/飞狐外传`：

- 《笑傲江湖》验证旧产物整体归档、物品候选不再无解释归零、对白覆盖和不可挪用抽样配额。
- 《飞狐外传》复验相同规则，确认实现没有书名、作者或固定实体特判。
- 中篇目标 60 分钟，约 50 章长篇目标 90 分钟；定向补漏单独计时。性能超时只记录证据，不放宽内容门禁。
- `run.json` 分别记录 AI 逐章、AI 合并/清理、定向补漏、脚本、人工等待和总耗时。

## 15. 风险与控制

- **AI 章节遗漏**：逐章文件、章节摘要强制一章一条、候选覆盖账本和缺口章节定向补漏。
- **跨章归并错误**：脚本先归组，AI 只处理歧义；无法唯一确认转人工。
- **跨 run 污染**：全部中间产物绑定 run 目录和 source 哈希；多活动 run 必须显式选择。
- **旧构建残留**：新 run 前严格归档，只保留原著、`ch_split/` 和 `_archive/`。
- **ID 震荡**：AI 不接触最终 ID；脚本一次生成和重写。
- **低等级人物输出膨胀**：后三档使用简略投影。
- **对白耗时膨胀**：逐章只为核心/重要事件保留短候选，最终每事件最多一条；覆盖率替代固定凑数。
- **数量凑数**：数量检查只报告；补漏由候选去向和覆盖异常触发，物品和普通动作准入规则优先。
- **崩溃后预算刷新**：预算绑定输入哈希并持久化。
- **接受产物被后续改写**：上游哈希链每步复验，篡改立即阻断。
- **归档误伤或丢失**：先清单、路径约束、同文件系统移动、失败回滚和归档后哈希复验。

## 16. 回滚

- 实现未启用前，删除独立 Skill 和新任务产物即可；现有 `generate-kb` 不受影响。
- 旧构建归档可按 `archive-manifest.json` 将所有条目恢复到原相对路径；恢复前必须验证目标路径为空或与清单一致。
- 安装失败时删除未生效的临时目录；旧构建仍在 `_archive/<timestamp>-before-<run-id>/`，不会被覆盖。
- 完整 run 归档保留所有已接受章节和补漏文件；需要修复时显式恢复该 run，不必重读未变化章节。

## 17. Skill 自主执行契约

用户调用 `generate-game-kb` 时只提供书籍目录，不负责选择或排列内部命令。当前主模型是唯一流程编排者，并按可观察状态自主路由：没有活动 run 时执行旧产物归档并创建 run；恰有一个活动 run 时恢复该 run；多个活动 run 无法唯一选择时停止并请求人工裁决。状态读取只发生在启动、上下文恢复或阶段边界，不能用循环轮询代替推进。

章节阶段的数据流固定为：主模型读取持久化并发上限与下一 attempt 并按章派发宿主原生子代理 → 子代理直接完整读取 run 中唯一一个章节原文 → 子代理写入主模型指定的 run 级 staging 路径并只返回路径 → 主模型串行调用 `accept` → CLI 校验 run/unit/attempt、持久化结果并消费 staging → 子代理上下文销毁。CTX/context-mode、搜索结果、摘要、启发式抽取和外部模型 CLI 都不是章节事实输入。主模型继续承担合并、清理、补漏和质量复核的语义判断；确定性脚本只负责归档、状态、限流退避、staging 生命周期、哈希、校验、投影、安装与报告。

主模型随后根据持久化状态依次完成章节、覆盖检查、条件式类别补漏、合并、引用检查、一次清理、最终构建、固定质量抽样、安装、已安装复验和 run 归档。章节子代理不能调用 `accept`、编辑共享状态、把正文回传主上下文或把任务再次委派。`manual_review`、多活动 run、宿主缺少原生子代理能力或无法唯一解析的引用是明确停点；不得让用户手工编排命令、自动重置单元或启动外部模型 CLI 绕过停止条件。
