# 重构武侠知识库生成与质量门禁

## Goal

建立一条以小说原文为唯一事实来源、面向武侠游戏资料复用的知识库生成流程。流程需要稳定召回主要角色、物品、地点、门派/势力和大事件，保留全部原文明确定名且可定位的武功与招式，并截取有代表性的原文对话；明显漏提取的产物不得通过自动门禁。

最终知识库应支持两类核心用途：

- 将武功、招式及其原文表现转化为游戏技能原型。
- 将性格鲜明的角色、标志性物品和代表性对话转化为彩蛋 NPC、剧情或世界观素材。

## Background And Evidence

- 当前 `generate-kb` 的定位是“LLM 先验生成，再以原文 locate/verify”，候选全集并非从原文系统枚举，因此 locate/verify 只能验证已想到的条目，不能发现未召回条目（`.agents/skills/generate-kb/SKILL.md`）。
- 当前流水线有 15 个阶段标题和至少 16 个命令或人工步骤，且 Phase 编号与执行顺序交错（`.agents/skills/generate-kb/pipeline.md:5`）。
- 《连城诀》初版只有 21 角色、3 门派、4 地点、5 武功、3 招式、4 物品、4 对话；人工返工后变为 35、3、10、12、3、12、60。说明现有自动流程不能稳定达到人工认可的召回规模（Trellis session `ce41b21b-ea90-4a62-b867-b2cca38410c7`）。
- 少于 20 章的作品只要求 5 个武功、3 个物品；数量得分按“通过的类别数”计算，4/5 类达标即得 80 分（`.agents/skills/generate-kb/scripts/assess-quality.js:837`）。完成门禁只要求该分数不低于 80，且完全不检查招式数量（`.agents/skills/generate-kb/scripts/assess-quality.js:982`）。这解释了《连城诀》初版为何能够通过。
- 对话的 `representativeness` 实际只是“有 speaker 的比例”；`characterFit` 和章节覆盖率虽被计算，却不进入完成门禁（`.agents/skills/generate-kb/scripts/assess-quality.js:441`、`.agents/skills/generate-kb/scripts/assess-quality.js:967`）。
- 当前《天龙八部》只有 37 条对话，覆盖 28% 章节，`characterFit=0`，章节摘要质量代理值为 2，verification report 的全量 grounded ratio 为 55.5%，但 `completion_gate_passed=true`、`honest_overall_score=87`（`金庸/天龙八部/reports/quality_report.json`）。
- 当前 baseline 被检测到角色 ID 与最终 `characters.json` 为子集/拷贝，仍因存在 relationships/events 而被标记为 `ok`；同源 baseline 可以继续参与评分（`.agents/skills/generate-kb/scripts/assess-quality.js:116`）。
- 《天龙八部》的物品链路存在 256 个 candidates、36 个 filtered、15 个 final，最终文件与 filtered 集合大幅分叉；候选中还混入武功和地点。六脉神剑图谱、玉像等强剧情物品未进入最终 `items.json`，说明候选分类、筛选和最终产物之间缺少可审计的保留/淘汰契约。

## Requirements

### R1. 原文唯一可信源

- 所有事实性字段、关系、事件、技能效果、招式描述和对话必须能回指原文证据。
- LLM 先验只能用于生成检索同义词或提出待验证候选，不能作为事实证据，也不能直接决定条目存在。
- 无法在原文定位的事实不得进入正式数据；不确定信息应进入审核队列，而非用合理推断补全。

### R2. 原文先行的召回机制

- 候选清单必须通过逐章扫描原文建立，覆盖角色、武功、招式、物品、地点、门派/势力、事件和对话八类目标。
- 对命名武学、明确招式名、反复出现或推动剧情的物品建立独立召回通道，不能只依赖一次通用 LLM 提取。
- 每个候选必须保留“发现位置、归类结果、保留/淘汰原因”，使 `candidate -> reviewed -> final` 可追踪。
- “主要”采用可审计规则判断，例如出场/提及频次、跨章跨度、是否参与关键事件、是否改变人物关系或剧情结果；不能仅凭固定数量上限截断。

### R3. 游戏资料导向的数据质量

- 武功与招式必须区分：武功是体系，招式是可被游戏技能化的具体动作/变化；明确命名的招式不能因只属于一个武功而从独立可查询数据中消失。
- 原文明确定名且可定位的武功和招式全部入库，不以“是否主要”为删除条件；使用 `importance` 标记核心、重要或冷门，供消费端筛选。
- 技能条目应保留原文可证的使用者、效果、限制、代价、克制或典型战斗表现；原文未说明的字段不得脑补。
- 物品应覆盖剧情钥匙、信物、秘籍、独特兵器、毒物/药物及有强人物辨识度的随身物，排除普通环境物件。
- 角色条目应优先保留可用于彩蛋设计的性格、说话方式、标志行为、关系转折和代表场景，并逐项引用原文。

### R4. 事件与对话联动

- 先从逐章事件表识别主要事件，再从事件原文窗口截取关键对话。
- 另设“人物特征对话”通道，选择能体现性格、口癖、价值观、关系或反差的原文对话。
- 每条入库对话必须标记选择目的、关联事件或人物特征、说话者、上下文窗口和精确原文位置。
- 对话质量不能用“原文中存在”代替“值得入库”；真实性、事件代表性、人物贴合度和覆盖分布必须分别验收。

### R5. 非循环质量门禁

- 被测数据、候选生成结果和质量基准不得由同一产物相互派生后互相打分。
- 没有独立金标时不得输出伪装成完整性结论的总分；应报告可直接观测的证据覆盖、未解释命名项、章节覆盖、候选淘汰和召回饱和度。
- 门禁必须包含逐类别硬失败条件；角色数量不能补偿武功、招式、物品或对话的严重缺失。
- `verification_result` / `verification_report` 的全量 grounded 指标、主事件/人物特征对话覆盖、对话分布、人物贴合、摘要质量和招式闭环必须参与完成判定。
- 旧版《连城诀》低数量产物和当前《天龙八部》产物必须被新门禁拒绝，并给出可执行的缺口报告。

### R6. 流程收敛

- 将当前多次交错的生成、修复、重跑和审核阶段收敛为少数职责明确、可恢复、幂等的阶段。
- 每个阶段只有一个权威产物和明确 gate，失败后从最近阶段恢复，不重复已通过工作。
- 最终八类数据保持统一 schema、统一 source reference 语义和统一报告格式。

## Out Of Scope

- 构建具体游戏或直接设计数值平衡。
- 收录原文中的每一个普通人名、普通物件或环境名词。
- 使用影视剧、百科、读者记忆或其他改编作品补充小说原文没有的设定。

## Acceptance Criteria

- [ ] 对每个正式条目，可从 source reference 精确定位到原文；抽样与自动校验均无外部设定混入。
- [ ] 各目标类别均有独立候选发现、审核和完成状态；任一关键类别严重不足时总门禁失败。
- [ ] 对命名武功、命名招式和剧情关键物品输出“原文候选、保留项、淘汰项及原因”差异报告。
- [ ] 所有原文明确定名且可定位的武功、招式均进入最终 `skills.json` / `techniques.json`，只分级、不删除。
- [ ] 《连城诀》旧版的 5 武功、3 招式、4 物品、4 对话产物无法通过新门禁。
- [ ] 当前《天龙八部》产物因对话覆盖 28%、`characterFit=0`、全量 grounded 55.5% 等问题无法通过新门禁。
- [ ] 门禁不再接受 self-referential baseline；无独立金标时明确输出 `no_gold`，不计算金标完整性总分。
- [ ] 对话验收同时覆盖真实性、事件代表性、人物特征贴合和章节/主事件分布。
- [ ] 新流程阶段数量显著少于当前 15 个阶段标题，且正常运行不需要临时代码或手工修章节号。
- [ ] 为召回、分类、证据校验、非循环门禁及两个回归案例提供自动化测试。

## Compatibility Decision

- 允许重构内部流程与中间产物格式。
- 最终知识库的八类数据消费接口保持兼容；现有知识库可按新流程重新生成。
