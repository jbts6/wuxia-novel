# 快速知识库生成流程

## Goal

在不修改现有 `.agents/skills/generate-kb` 的前提下，新增一套独立、简单、以原文逐章理解为核心的知识库生成流程，为未来武侠游戏的功法设计、剧情原型、角色彩蛋、世界观和标志性物品提供可靠素材。该流程优先缩短人工等待时间并稳定达到约 95 分的可用质量，不把 token 消耗作为优化指标，也不追求穷尽所有长尾实体或达到 100 分。

## Background

- 当前 `generate-kb` 使用六阶段受管状态机、claim/submit work item、多重审计门禁和原子发布流程；一次长篇生成通常需要两到三小时或更久。
- 现有安全审计确认该状态机存在确定性自循环、不可恢复阻塞、无限重试和崩溃恢复缺口。新流程不得复用这些状态机和发布指针机制。
- 用户早期验证过的有效方法是：逐章阅读并提取，集中整合、过滤和去重，再排除分类错误及描述词伪实体。
- token 成本不计入成功指标；允许 AI 直接阅读完整章节并为理解原文消耗更多上下文。
- 知识库的主要消费场景是未来武侠游戏设计，而非一般文学百科；数据选择和描述应优先保留可转化为玩法、剧情、角色彩蛋与世界观素材的经典元素。
- 《笑傲江湖》首次快速试跑暴露出三类通用缺口：旧构建副产物没有完整归档、7 个逐章物品候选在合并后归零、对白候选只覆盖少量前部章节且最终抽样没有实际检查物品和对白。修订不能为单书写特判，必须把这些问题转化为跨书籍的运行隔离、候选去向和类别覆盖契约。
- 2026-07-15 边界决策：本任务保留快速流程第一阶段的实现、自动化合同与 whole-book 试跑负向证据；类别语义决策、确定性整书组装及 fresh 两书正向验收迁移到 `07-15-game-kb-deterministic-assembly`。本任务归档不代表旧 whole-book merge/clean 已满足新版正向验收。

## Requirements

- 保留现有 `.agents/skills/generate-kb` 的文件、命令、状态和产物行为，不在其目录内做任何修改。
- 新流程命名为 `generate-game-kb`，实现位于独立的 `.agents/skills/generate-game-kb/`。
- 新流程必须拥有独立的 Skill 名称、目录、命令入口和按 `run-id` 隔离的运行产物，不能读取或写入现有 managed run、`.kb/current` 或正式发布指针。
- 最终产物保持与现有应用兼容的九类 JSON 文件名、顶层结构、ID 规则和交叉引用约定，并以 `<小说目录>/data/` 作为最终落点。九类为人物、事件、物品、功法、招式、门派/势力、地点、对白和章节摘要。
- 生成主流程保持为三个可理解阶段：逐章阅读与提取、全书整合与去重、错误分类与伪实体清理。正常作品只执行这一遍主流程；确定性覆盖检查发现异常时，才插入受影响类别的一次有界补漏，不能固定要求每本书执行第二遍全书生成。
- 采用分章文件型流程：主模型为每个待处理章节启动一个隔离的原生子代理；每个子代理只直接完整读取一个章节并写出独立草稿。`prepare` 在新建或恢复 run 时都必须创建 `.game-kb-work/runs/<run-id>/staging/`。已完成且输入哈希未变化的章节直接跳过，全部章节完成后再统一合并、清理和校验。每次运行的 source 快照、staging 草稿、提交归档、接受产物、报告、日志、进度和人工问题必须全部绑定到该 run。
- 主模型拥有唯一的流程控制权并串行提交章节草稿；章节子代理可以并行读取互不依赖的章节，但不能调用 `accept`、修改进度或写 accepted 产物，因此不需要 claim、lease、事件日志或阶段互斥状态机。
- 章节子代理采用 run 级持久化自适应并发：新 run 从 10 开始，实际派发数不超过宿主可用槽位和待处理章节数；同一批一个或多个 429 只触发一次 `10 → 5 → 2 → 1` 退避。429 不消耗语义提交次数，恢复或上下文压缩不重置上限，新 run 才重置；并发为 1 仍遇到新 429 时停止并报告外部限流。
- 用户只需调用 `generate-game-kb` Skill 并提供书籍目录；当前主模型必须根据根目录、活动 run 和持久化单元状态自主决定归档、恢复及后续阶段，不得要求用户指定内部脚本或命令顺序。
- 逐章语义提取必须由隔离的原生章节子代理直接完整读取章节原文。CTX/context-mode、检索摘要、关键词启发式和外部模型 CLI 只能用于非事实统计或不得参与语义生成，不能代替章节子代理阅读原文。子代理只向主模型返回 run 级 staging 草稿路径和状态，不回传原文或 JSON 正文。
- 新建 run 前必须先归档旧构建产物。归档成功后的小说目录根级条目只允许存在唯一原著 `.txt`、`ch_split/` 和 `_archive/`；其他现有条目均按原相对路径移动到带时间和来源标识的归档目录，不得删除。
- 归档必须先生成完整 `archive-manifest.json`，再执行同文件系统移动并在失败时回滚；多个原著文本、越界符号链接、路径冲突或部分移动失败均须 fail closed。归档完成后必须重新扫描目录并验证允许列表。
- 原著 `.txt` 是唯一事实输入。`ch_split/` 只有在章节数、标题和逐章内容哈希与从当前原著确定性切分的结果完全一致时才可复用；旧 `build/`、`data/`、`reports/`、`prompts/`、`review/` 和 `.game-kb-work/` 不得成为新 run 的生成输入。
- 逐章提取至少覆盖人物、事件、物品、功法、招式、门派/势力、地点和章节摘要；所有事实只以小说原文为依据。
- 九类数据的召回优先级偏向游戏素材价值：功法/招式最高，经典事件桥段、鲜明人物与标志性物品次之，门派/势力和代表性地点用于世界观支撑；普通日常细节保持低优先级。
- 功法与招式应偏高召回，重点保留规范名称、所属体系、使用者、效果、限制、修炼或使用条件、克制关系、阶段变化和代表性表现，便于未来提炼战斗系统原型。
- 原文明确定名且能够与普通动作区分的功法和招式原则上全部保留，冷门、低频或威力弱只降低重要性，不作为删除理由；“拍出一掌”“挥剑刺去”等无专名普通动作不得凑成招式。
- 不新增“动作描写”类别。无正式名称的普通攻击、移动和姿态描写全部排除；仅当动作能说明某项已命名功法或招式的独特机制时，才可概括进对应记录的 `combat_style`、`effects` 或描述字段。
- 事件应优先保留可识别的经典桥段、关键冲突、反转、奇遇、传承、复仇、门派斗争和人物关系转折，并记录起因、关键过程、结果及跨章范围，便于未来提炼剧情原型。
- 人物除主角和主要配角外，还应保留名字、绰号、能力、行为或经典场面足够鲜明、适合作为游戏彩蛋的次要人物；无名且无辨识度的路人仍可忽略。
- 人物沿用核心、重要、次要、龙套、背景五级。只有核心和重要人物生成完整生平、详细性格、关系变化、功法关联和多章依据；次要、龙套和背景人物全部使用简略记录。
- 简略人物仍保留兼容字段，但内容只要求姓名/别名、身份、等级、简短定位、代表性行为或彩蛋价值、必要的功法/关系引用和章节依据；无信息字段使用空字符串、空数组或空对象，不得为满足旧详细模板而编造长传记或五项性格特征。
- 标志性物品必须保留准确原名、类别、特殊效果、持有者/使用者、获取或流转过程和剧情作用，使经典毒药、丹药、秘籍与神兵能够直接被检索。
- 游戏素材偏向只影响选择、重要性和描述重点，不能为了游戏化改写原著名称、效果或事实。
- 九类事实数据之外，额外生成 `reports/game_materials.json` 轻量索引。索引只引用九类数据中的稳定 ID，并按战斗系统原型、经典剧情桥段、角色原型/彩蛋、标志性物品、门派与世界观素材分类。
- 游戏素材索引可以记录推荐用途、辨识度和基于原著事实的简短理由，但不得复制出第十类实体、创造游戏设定或反向修改事实数据。
- 对白降级为事件的附属资料，但逐章阶段不能把“最终每事件最多一条”误用成提取上限。对核心/重要事件可以提取多条短候选，并记录事件重要性、是否存在可引用对白及来源章节；最终才为每个事件最多选择一条。
- 最终对白只保留能代表重要事件、关系转折或经典人物形象的短句，每条必须关联 `event_id` 和正确章节。所有可引用的核心/重要事件都必须有对白或明确的省略理由，最终覆盖率低于 70% 时触发一次对白专项补漏。
- 可引用核心/重要事件分布在至少 8 章，而对白候选覆盖的这些章节少于 30% 时，视为章节分布异常并触发同一次对白专项补漏预算，不能为此另开第二轮。
- 合并时处理别名、同名异写、重复事件、跨章实体延续和引用关系，避免把同一对象拆成多个实体。每个逐章候选都必须在覆盖账本中得到 `merged_to`、`rejected` 或 `ambiguous` 去向；高价值候选不能无解释消失。
- 清理阶段必须识别至少以下错误：武器被当作招式、功法与招式混淆、普通描述词被当作专名、泛称被当作人物或组织、无原文依据的实体。
- 原文证据的验收精度以“章节正确”为准，不要求行号、段落或引用文本完全准确；`source_refs` 保持兼容字段，但行段只作为近似定位信息。
- 同一事件可以跨越多个章节，必须允许一个事件关联多个章节证据；不要求把跨章事件压缩为单一章节，也不要求枚举每个过渡章节。
- 每类产物都必须给出基于章节数和作品篇幅的建议数量区间、实际数量和单位章节密度。数量只作报告和异常信号，不能覆盖准确性，也不能单独驱动 AI 凑数或删数。
- 定向补漏由证据覆盖异常触发，而不是由固定数量区间触发。已有候选在合并或清理后归零、候选去向不完整、重要事件对白覆盖不足、章节分布异常、抽样缺类或引用断裂时，才允许对受影响类别执行一次语义补漏；不得重跑全书提取、全书合并或第二次全量清理。
- 正文汉字数不少于 15 万，或核心/重要事件不少于 20 个的作品，若物品候选为 0，必须执行一次物品专项复核。复核可以确认作品确实没有符合准入标准的重要物品，但必须留下章节范围和结论记录，不能静默以空数组完成。
- 物品只收录具备明显知识价值的对象：神功秘籍、对剧情有重要推动作用的关键物、较高级或具特殊作用的毒药/丹药、神兵利器及其他稀有特殊物品。
- 普通随身武器、日用品和不推动剧情且无特殊能力的物品不收录，即使它属于主角；不能为了达到数量区间降低物品准入门槛。
- 质量目标是约 95 分的实用水准；长尾召回不足可以接受，但核心人物、主线事件、关键关系和类型正确性不能依靠人工逐条返工。
- 优先优化总耗时和执行稳定性，不优化 token 消耗。
- 以约 50 章的完整长篇为基准，端到端目标耗时不超过 90 分钟；该指标用于试跑和方案比较，不因外部模型或机器偶发变慢而使内容产物失效。
- 每个阶段必须可从同一 `run-id` 的已有文件继续，已完成章节不得因单章失败而全部重跑；任何命令都不能从另一 run 或共享根目录自动拾取中间文件。
- AI 修正失败必须有界：章节、合并、清理、`recall:<category>` 和 `supplement:<category>` 单元各自最多提交 3 次。定向补漏只允许一次语义生成，其余预算只能修正格式或结构错误；达到上限后生成结构化问题清单并进入人工处理终态。
- 尝试次数、输出哈希和标准化错误指纹必须持久化，重启流程不能重置预算。输出无变化、同一错误重复或出现 `A → B → A` 状态震荡时，应在达到上限前判定无进展并停止。
- 每份 AI 提交必须来自当前 run 的 `<run-dir>/staging/<unit>_attempt_<attempts+1>.json`。`accept` 一旦记录提交，无论校验成功或拒绝都必须消费该 staging 文件；编号较小的残留文件不得再次提交，也不得改变 `progress.json`。
- 任何已接受的章节、合并、清理或补漏产物都必须进入哈希链并保持不可变。后续阶段发现已接受文件被改写时，必须以 `ACCEPTED_ARTIFACT_MUTATED` 阻断，不能重新计算哈希后继续。
- AI 不负责生成、修补或反复调整最终全局 ID 索引。合并完成后由确定性脚本一次生成稳定 ID，并统一重写九类 JSON 的交叉引用；无法解析的引用进入人工清单，不能交给 AI 循环改 ID。
- 某个任务进入 `manual_review` 后，流程继续处理其他互不依赖的章节或批次；所有问题最终合并为一份人工清单，并在写入 `data/` 前停止。
- 自动校验只做确定性、低成本且能显著提高质量的检查；不得重新引入多阶段互斥门禁或复杂事件状态机。
- 最终内容质量只执行一次加权抽样：标准样本 40 条，其中功法/招式 12 条、事件 8 条、人物 5 条、物品 5 条、对白 4 条、门派/地点 4 条、章节摘要 2 条。类别配额不可被其他类别顶替；某类合法为空时必须抽查对应的空库复核结论。
- 抽样记录以名称、类别、关键事实和引用章节正确为通过，不要求行号或段落精确；至少 38/40 条通过才达到 95 分。抽样低于 95 分直接进入 `manual_review`，不得自动触发全库返工。
- 全部校验通过后自动把完整结果写入 `<小说目录>/data/` 并写入当前消费者需要的 `<小说目录>/reports/`。安装结果复验通过后，完整 run 依据 `artifact-manifest.json` 归档到 `_archive/generate-game-kb/<run-id>/`，不能只归档若干最终 JSON。
- `run.json` 必须记录 `started_at`、`installed_at`、`archived_at`、各阶段耗时、补漏耗时和总耗时，以机器记录代替通过文件修改时间反推。
## Quantity Guidelines

数量区间按正文汉字数分档，仅用于一次性异常复核，不是硬门禁：

| 类别 | 短篇 `<15万字` | 中篇 `15–50万字` | 长篇 `>50万字` |
|---|---:|---:|---:|
| 人物 | 10–35 | 25–75 | 45–120 |
| 事件 | 20–80 | 60–220 | 150–400 |
| 重要物品 | 3–12 | 6–20 | 10–30 |
| 功法 | 5–20 | 10–35 | 15–50 |
| 招式 | 0–20 | 5–45 | 10–80 |
| 门派/势力 | 2–12 | 5–25 | 10–40 |
| 地点 | 5–25 | 10–50 | 20–80 |
| 代表性对白 | 5–25 | 20–60 | 40–100 |
| 章节摘要 | 每章 1 条 | 每章 1 条 | 每章 1 条 |

## Acceptance Criteria

- [x] 现有 `.agents/skills/generate-kb` 的 git diff 为空。
- [x] `archive-existing` 成功后，小说目录除唯一原著 `.txt`、`ch_split/` 和 `_archive/` 外没有任何根级条目；所有旧 `data/`、`reports/`、`build/`、`prompts/`、`review/`、旧工作目录及其他残留都保留在带清单的归档中。
- [x] 归档在路径冲突、越界符号链接或注入的中途失败下不会丢失文件，并能回滚到归档前状态；归档清单与实际文件哈希一致。
- [x] 新流程可以从一本包含原文的小说目录启动，并按章生成可检查的中间提取结果。
- [x] 两个并存 run 的草稿、进度、报告和 final 文件完全隔离；CLI 不会从错误 run 或小说根目录读取中间产物。
- [x] 中断后重启会跳过输入哈希未变化且已完成的章节，只处理缺失或失效章节。
- [x] 恢复时只接受持久化 `attempts + 1` 对应的当前 run staging 文件；重复提交已经消费的旧 attempt 会在写进度前失败，尝试次数保持不变。
- [x] 合并结果不存在重复 ID、悬空引用或同一实体的明显重复记录。
- [x] 分类清理能用固定测试样例拒绝武器/招式混淆、描述词伪实体和无证据实体。
- [x] 每类结果都输出目标数量区间、实际数量、单位章节密度和是否异常；异常只进入一次复核，不触发无限补提取。
- [x] 数量异常复核完成后，即使最终数量仍在建议区间外，也会记录解释并继续流程，不会再次要求补数或删数。
- [x] 证据检查能接受行段不精确但章节正确的记录，并拒绝只引用错误章节的记录。
- [x] 跨章事件可以保留多个不连续章节引用，且不会因存在多个章节被错误去重。
- [x] 物品测试集会保留秘籍、剧情关键物、高级毒药/丹药和神兵利器，并排除无特殊性、无剧情作用的普通随身物品。
- [x] 每个逐章候选在合并和清理覆盖账本中都有唯一去向；“7 个物品候选合并后变成 0 且无拒绝理由”的回归样例必须阻断并生成 `recall:items` 或 `supplement:items`。
- [x] 正文不少于 15 万汉字或核心/重要事件不少于 20 个的作品，物品候选为 0 时会执行一次专项复核；专项复核可以有证据地确认 0，但不能静默跳过。
- [x] 抽样验收能检索到作品中具有代表性的功法/招式、经典桥段、鲜明次要人物和标志性物品，并保持原著名称与事实不被游戏化改写。
- [x] 核心和重要人物具备完整详细信息；次要、龙套和背景人物保持字段兼容但内容明显精简，不会生成模板化长传记或凑足固定数量的性格词。
- [x] 原文明确定名的功法和招式具有高召回，同时不会把无专名的普通攻击动作生成成独立招式。
- [x] 九类最终数据及游戏素材索引中不存在独立的普通动作条目，也不会出现第十类“动作描写”文件。
- [x] `reports/game_materials.json` 中的每个条目都能解析到九类 JSON 的现有 ID，覆盖五种素材类型，且不会引入事实数据中不存在的实体。
- [x] 每条最终对白都引用一个现有事件和正确章节，任一事件最多一条；逐章阶段允许多个候选，最终可引用的核心/重要事件对白覆盖率不低于 70%，且不会为达到数量区间补充普通台词。
- [x] 对白只集中在少量前部章节而重要事件遍布全书的回归样例会触发 `recall:dialogues`，补漏只读取缺口章节和相关事件。
- [x] 同一章节或同一错误连续失败达到上限后，状态明确停在 `manual_review`，并输出失败对象、错误原因、尝试次数和建议动作。
- [x] 不存在 `next_action` 只能再次产生相同只读结果的自动循环；每个自动动作要么产生进展，要么消耗有限预算并终止。
- [x] 任一 AI 任务总提交次数不超过 3，进程重启后次数不会清零；测试覆盖相同错误重复、相同输出重复和 `A → B → A` 震荡三种提前终止场景。
- [x] 最终 ID 由脚本稳定生成并一次性重写引用；测试中不会因两个引用错误在不同 ID 方案之间反复修正。
- [x] 单个失败任务不会阻断其他独立章节处理，但任何未解决的 `manual_review` 都会阻止自动覆盖 `data/`，并生成一份汇总问题清单。
- [x] `manual_review` 不会被模型自动执行 `reset-unit` 绕过；只有用户明确授权后才能重置指定单元。
- [x] 最终产物通过 JSON 解析、最小 schema、引用完整性、原文证据存在性和分类规则检查。
- [x] 最终质量报告包含不可挪用类别配额的 40 条抽样结果，至少 38 条通过；物品和对白不能因配额重分配而缺席，合法空类由空库复核结论占位。
- [x] 最终抽样失败只生成 `manual_review` 问题，不触发新的自动生成、合并或清理循环。
- [x] 通过校验的九类 JSON 可完整写入 `<小说目录>/data/`，无需 `.kb/current`、bundle manifest 或额外发布服务即可被现有消费者读取。
- [x] 安装结果复验后，整次 run 的 source 快照、草稿、接受产物、报告、日志、进度、人工问题和 final 数据全部进入 `_archive/generate-game-kb/<run-id>/`，归档后没有活动工作目录残留。
- [x] `run.json` 给出可直接读取的开始、安装、归档时间和各阶段/总耗时，无需依赖文件修改时间估算。
- [x] 使用一部已有小说进行端到端试跑，记录实际耗时和质量抽样结果，以验证“明显快于当前两到三小时流程”和约 95 分目标。
- [x] 约 50 章长篇的后续基准目标保持为不超过 90 分钟；执行该基准时必须单独记录 AI 处理、脚本处理、定向补漏和人工等待时间。
## Out Of Scope

- 修复或重构当前 `generate-kb`。
- 追求 100% 长尾召回、零人工复核或完备的事务型发布系统。
- 使用百科、影视改编或模型先验作为事实来源。
- 修改 `.kb/current` 或引入现有 managed publish/bundle 机制。
- 建立通用攻击、移动、姿态或战斗动画动作库；如未来需要，应另设下游派生流程。

- Lightweight tasks can remain PRD-only.
- For complex tasks, add `design.md` for technical design and `implement.md` for execution planning before `task.py start`.
