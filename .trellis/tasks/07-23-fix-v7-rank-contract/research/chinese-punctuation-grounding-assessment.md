# 中文排版符号与逐字证据门禁核验

## 观察

《血海飘香》重跑中，多名章节 Worker 在最终写入前报告修正了引号、冒号、引文前缀或逐字措辞。Controller 正式拒绝一次：`chapter:015 attempt_01` 的 `items[6].source_refs[0].text` 返回 `SOURCE_QUOTE_NOT_FOUND`。

用户在本次重跑前已将原文中的中文双引号全量替换为英文双引号；这只能消除引号字形差异，不能统一其他中文排版符号。对本次 run 的 27 个 prepared chapter 统计后，正文仍同时包含 2,436 个中文句号 `。` 和 2,240 个英文句点 `.`，另有 88 个破折号 `—`、138 个顿号 `、`。正文因此不是单一标点体系，Worker 将句号等字形自动统一时仍可能产生逐字匹配失败。

当前持久化的 28 份章节草稿只保留 Worker 自检后的写入结果及一次正式重试。逐条比对其中所有 `source_refs[].text` 后，只有 `chapter:015 attempt_01` 留有不匹配，且它属于下文所述的内容改写；Worker 写入前自行修正过的纯符号失败未保留为可重放 artifact。因此，是否新增符号折叠不能用“正式拒绝只有一次”来否定，但实现必须先补充合成固件和可审计的真实样例。

## 现有行为

`grounding.js` 的 `normalizeEvidenceText()` 已执行：

- BOM / CRLF 归一化；
- Unicode NFKC；
- 连续空白折叠。

因此全角 ASCII 兼容字符通常已归一化，例如 `，` / `：` / `？` 可折叠到 ASCII 形式。NFKC 不会把所有中文排版符号等同，例如弯引号 `“”`、书名式引号 `「」` 与 ASCII `"` 仍不同，中文句号 `。` 与 `.` 也不同。

## chapter:015 不是纯标点误差

拒绝目标为：

```text
楚留香微一沉吟,道:"令尊入关前所接的那封书信,不知你是否瞧见过?不知那信上写着的究竟是什么?"
```

程序移除全部标点和空白后，仍无法在章节正文中找到相同词序；与正文最长连续词汇片段只有“的究竟是什么”。正文实际附近内容是楚留香请求黑衣少年说明信上内容，措辞不同。该错误包含模型改写或拼接，不应因“中文符号等价”而放行。

## 安全边界

不能采用“忽略全部标点后匹配”或编辑距离近似匹配。这会把拼接、删句、改写和错误归因伪装成原文证据，违反逐字 grounding 门禁。

可讨论的最窄机制是 allowlist typography folding：

1. 只折叠明确列举的一对一排版符号，不删除标点；
2. 除 allowlist 符号外，全部汉字、数字、字母及其顺序必须完全一致；
3. 规范化后必须在当前章节唯一命中，否则拒绝；
4. accepted YAML 的 `source_refs[].text` 必须替换为命中的源文本，而不是保留 Worker 的变体；
5. 记录 normalization audit，说明输入符号、源符号和规则；
6. 以 `chapter:015` 的错误目标作为 false-pass 固件，证明实际措辞改写仍返回 `SOURCE_QUOTE_NOT_FOUND`。

## 任务边界建议

该机制会修改 grounding、accepted projection、审计记录和测试，和 rank/level 合同是两个独立数据边界。建议建立独立子任务，先用真实失败固件证明“纯符号差异可恢复、措辞改写仍失败”，再决定是否启用；不要把它塞进 rank 修复提交。
