# generate-game-kb 成本效率评审核验

## 结论

`docs/generate-game-kb-cost-efficiency-review.md` 有价值，但更适合作为“待验证的优化假设清单”，不能直接当实施路线图。静态合同重复传输的判断有本次 run 数据支持；删除 Worker 递归自检、双章打包、扩大并发窗口以及 50–60% 总节省则缺少端到端计量，部分结论与当前 v7 规范或实际运行证据冲突。

## 实测基线

样本：`run-2026-07-23-xuehai-v3-rerun` 的 27 个 attempt 1 job input。

| 指标 | 实测 |
| --- | ---: |
| job input 总字节 | 614,499 |
| 重复 worker contract | 121,959（19.8%） |
| 重复 taxonomy | 13,338 |
| contract + taxonomy 占比 | 22.0% |
| 章节正文 | 392,045 |
| 实际重试 | 1/27；`chapter:015` 为逐字引文错误 |
| 仓库内显式 prompt cache 支持 | 0 处 |

字节占比支持“静态前缀约占输入两成”的方向，但不能直接等同于计费 token、缓存命中率或最终账单节省。

## 分项判断

### P0-2 静态合同与 taxonomy 缓存

有价值，且是文档中证据最强的一项。27 章样本的静态部分占 job input 22.0%。但当前仓库只生成自包含 JSON job，不拥有模型 API dispatch adapter，也没有 `cache-control` / prefix-cache 实现。是否命中缓存取决于宿主派发层，不能只改 generate-game-kb 代码就承诺“零风险省 20%”。应先增加实际 token / cached-token 观测或确认宿主能力。

### P0-1 删除 Worker 递归 preflight

“Controller 必须独立复验”正确；“Worker 自检完全没有额外价值”不成立。本次运行中多个 Worker 在写后自检阶段修正逐字引文、引号和名称证据，减少了 Controller 拒绝后的整章重试；`chapter:015` 仍漏过一条并实际产生 attempt 2。去掉自检会节省单次推理时间，但可能提高昂贵的语义重试率。当前应保留，并先记录“自检修正数、接收拒绝率、重试 token”再决定是否裁剪；可以研究缩短为针对逐字证据和字段形状的定向检查，但不能直接删除。

### P1-1 双章打包

收益假设合理，但“质量不变”没有证据。当前 Trellis 规范明确每个 job 恰好一个章节、失败只重跑该章、固定五章窗口；双章打包会扩大失败域、改变 output / retry / accepted 原子边界，并增加 Controller 拆分和中断恢复复杂度。它是合同架构变更，不是低风险降本项，应在取得真实每调用固定开销后另立任务。

### P1-2 确定性机械修复

方向有条件价值，但文档例子部分沿用旧合同：v7 Worker 不要求 `LOCAL_KEY_REQUIRED`，Controller 会注入 local key；删除 forbidden 字段或补缺数组也未必能保持语义。适合确定性处理的范围应只包含可证明字节等价或语义不变的 YAML 表示问题。当前 run 没有机械 repair 样本，不能支持“机械失败占多数”的结论。

### P2-1 窗口 5 提高到 8/10

只可能改善墙钟，不降低模型 token。本次宿主最多同时运行 3 个章节 Worker，窗口 5 已高于实际并发；直接提高常量不会形成 8/10 个并发调用。当前规范还明确固定五章窗口、整窗 accepted 后再签发下一窗。若要优化墙钟，应先测量窗口尾部空闲，并比较“扩大固定窗口”与“改变为滚动补位”的恢复与审计成本；后者当前被规范明确禁止。

### P2-2 语义失败局部修复

有现实价值，优先级可能高于文档给出的 P2。本次 `chapter:015` 只有一条引文标点错误，却重新生成整份章节 YAML。可以另行研究把 rejected draft 与 `previous_errors` 一并提供给同一 chapter Worker，要求保持其他语义字节稳定；仍需 Controller 全量复验，且不能让局部 patch 绕过 accepted 原子性。

### P2-3 过渡章轻量化

文档“不建议默认开启”的结论正确。它直接触碰逐章穷举与召回要求，不应作为通用降本方案。

## 建议重排

1. 先增加可比较的 token、缓存命中、Worker 自检修正、拒绝与重试观测。
2. 在确认宿主支持后验证静态 prefix caching；不改落盘的自包含 job 合同。
3. 单独设计语义 attempt 2 的局部修复输入，使用实际 `chapter:015` 作为回归样本。
4. 只对有严格语义不变证明的 YAML 表示错误做确定性修复。
5. 双章打包、窗口扩大或滚动补位必须作为独立合同变更评审，不与 rank 修复捆绑。

因此，原文“预计降低 50–60%”目前不能作为承诺：各项节省存在重叠，缺少缓存命中和总 token 账单，也没有计入打包失败域、自检删除后的重试增量。
