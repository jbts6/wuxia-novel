# generate-kb 后续优化计划

## 总体判断

当前流程已经具备原文证据、候选账本、独立查漏和 G1-G5 硬门禁。下一阶段不优先增加知识库字段或更多总分，而是依次解决：

1. 全书扫描难以稳定跑完。
2. 门禁失败后返工定位不够直接。
3. 候选归并仍有大量重复劳动。
4. 缺少可衡量真实召回率的人工基准。
5. 在质量可测之前，无法安全优化模型成本。

执行顺序固定为：**稳定执行 -> 精确返工 -> 辅助归并 -> 质量标定 -> 选择性复核 -> 成本优化**。

## P0：可恢复的一键执行器

目标：把当前由执行者逐窗口驱动的流程收敛为一个可中断、可恢复的正式入口。

建议接口：

```bash
node scripts/run-pipeline.js <novel-dir> [--resume] [--stage <name>] [--concurrency <n>]
```

必须具备：

- 根据 source hash 和 `scan-manifest.json` 识别未完成窗口。
- 调度 named-inventory、event-dialogue 和 gap-audit，不改变各 pass 的输入隔离规则。
- 支持并发上限、限流、超时、指数退避和可配置重试。
- 每个模型结果写入前立即进行 JSON、schema、window ID 和 source_ref 基础校验。
- 先写独立 partial，再原子合并到 `candidates.jsonl`，避免中断造成半行或账本损坏。
- 同一 window/pass 可安全重跑，不产生重复候选或重复完成记录。
- 保存模型、prompt 版本、开始/结束时间、token、费用、重试次数和失败原因。
- 在 Stage 3 人工归并、Stage 4 新候选回流和最终门禁前设置明确检查点。
- 输出机器可读进度和人读摘要，包括完成率、失败窗口、预计剩余工作量。

完成标准：

- 任意时刻中断后，使用 `--resume` 能从未完成窗口继续。
- 重跑已完成批次不会改变候选数量或制造重复 decision。
- 单个窗口永久失败不会阻塞其他窗口，最终会进入明确的失败队列。
- 至少用一部长篇小说完整跑通三类扫描及恢复演练。

## P1：门禁驱动的返工闭环

目标：门禁不仅报告 PASS/FAIL，还直接生成下一轮可执行的工作清单。

新增统一问题结构，至少包含：

- `gate_id`
- `issue_type`
- `severity`
- `chapter` / `window_id`
- `candidate_id` / `final_id` / `event_id` / `dialogue_id`
- `evidence`
- `suggested_action`
- `rerun_stage`

需要支持的返工类型：

- 漏扫窗口重新扫描。
- 无 decision 候选重新归并。
- source_ref、章节号或引文不匹配的定点修复。
- 缺少事件对话、人物特征对话或语义豁免的定点补充。
- field_source_refs 缺失或 weak/unverified 的字段重做。
- gap audit 新候选回流 Stage 3 后，只重跑受影响的检查。

完成标准：每个阻塞 G1-G5 的错误都能追到具体对象，并给出明确的下一步处理动作。

## P2：候选归并助手

目标：减少逐条阅读 `candidates.jsonl` 的重复劳动，同时保留人工可审核的最终决定。

先使用确定性规则生成候选簇：

- 规范化名称完全相同。
- 名称与 alias 交叉命中。
- 同一或相邻窗口中的重复 source_ref。
- 同名但 skill/technique 分类冲突。
- 同一实体跨窗口出现且描述互补。

输出 `build/reconciliation-suggestions.jsonl`，只提供建议，不直接替代 `decisions.jsonl`。低风险重复可以批量确认；名称冲突、类别冲突、证据不足和一对多映射必须进入人工或模型复核队列。

约束：

- 不得自动 reject 原文明确定名的武功或招式。
- 不得仅凭字符串相似度合并不同实体。
- 每个建议必须列出全部 candidate IDs、证据窗口和建议理由。
- 正式 keep/merge/redirect/reject 仍需写入可追踪 decision。

完成标准：在不降低账本闭环率和分类准确率的前提下，显著减少需要独立审核的候选簇数量。

## P3：跨书人工金标与质量标定

目标：从“流程没有发现更多遗漏”升级为“能够测量真实召回和错误率”。

建立小规模、人工维护、source hash 绑定的评测集，至少覆盖：

- 金庸、古龙、梁羽生等不同文风。
- 长篇、短篇、多卷合订和特殊章节编号。
- 八类最终数据。
- 低频命名武功、明确招式、易混淆物品和别名人物。
- 主要事件对话与核心人物特征对话。

按类别报告：

- 候选召回率。
- 最终实体精确率与召回率。
- skill/technique 分类准确率。
- alias 归并准确率。
- source_ref 和 dialogue 原文命中率。
- 主要事件、核心人物的语义覆盖率。

完成标准：prompt、模型、窗口或归并规则的修改必须在同一金标集上对比；没有基准结果时，不得宣称召回率提升。

## P4：风险驱动的第二审阅者

目标：只把高风险内容交给多 reviewer 或异模型复核，避免全书固定增加数倍成本。

高风险条件包括：

- 最终 gap round 仍产生新候选。
- skill/technique 或实体类别存在冲突。
- 多个候选可能对应多个最终实体。
- source anchor 过短、过泛或缺少明确动作与实体。
- 主要事件只有一条边缘相关对话。
- 核心人物的 persona 对话缺少清晰 trait 证据。
- 语义豁免理由过于笼统。

复核结果必须落入结构化 review ledger，记录同意、反对、修订建议和最终裁决，不允许只输出自然语言报告。

完成标准：在人工金标上证明风险路由能发现单模型遗漏，并明显低于全量多模型复核成本。

## P5：证据质量与引用修复

目标：提高 source anchor 和复杂描述字段证据的可读性、区分度与稳定性。

- 对 anchor 做质量分级：是否包含明确实体、具体动作、足够上下文和可区分信息。
- 低质量 anchor 进入重写队列，但不得改变事实或脱离原文。
- 提供正式的章节引用修复工具，修复前后都校验 source hash 和引文命中。
- 为 field_source_refs 检查“字段结论是否真的由该证据支持”，不只检查字符串存在。
- 增加事件与人物对话覆盖矩阵，避免“每项有一条”掩盖代表性不足。

完成标准：grand weak/unverified 保持为 0，低质量 anchor 比例下降，引用修复过程完整可追踪。

## P6：成本与吞吐优化

前置条件：P0-P3 已完成，且有稳定的人工评测基线。

可以比较：

- 初扫使用较低成本模型，歧义与高风险内容升级到更强模型。
- 不同窗口大小和 overlap 对召回率、重复率和调用量的影响。
- prompt 精简、结构化输出和批处理对 token 的影响。
- 并发度、缓存和失败重试策略对总耗时的影响。
- named-inventory 与 event-dialogue 是否存在可安全共享的预处理结果。

任何成本优化必须同时报告质量变化。若武功/招式召回、对话真实性或 G1-G5 任一项退化，则不得采用。

## 暂不优先

- 增加新的实体类别或大量可选字段。
- 用可补偿总分替代 G1-G5。
- 为提高数量而放宽原文证据要求。
- 在没有人工基准前全量更换模型或缩减扫描轮次。
- 全书固定使用三 reviewer 或异模型复核。
- 从旧 data 或 baseline 反向生成候选，造成自指评测。
