# generate-kb 质量指标体系方案

## 问题分析

当前 generate-kb 的质量校验覆盖三层：
1. **技术正确性**（locate.js / verify.js）：source_ref 能否定位到原文、子串是否匹配
2. **结构正确性**（cross-validate.js）：跨 JSON 的 ID 引用完整性、关系对称性、幻觉检测
3. **主观质量**（review-all.md）：LLM 审阅者挑刺（跨书混淆、说话风格、时代背景）

**缺失的维度**：内容完整性（是否遗漏重要实体）、内容准确性（描述是否正确）、覆盖率（高频提及是否被 KB 覆盖）。

## 核心思路：AI 凭书籍认知订指标

**可以，而且应该这样做。** LLM 对金庸/古龙/梁羽生/黄易的经典作品有丰富先验知识——知道每本书的核心角色、关键事件、重要武功、门派体系。这些知识可以转化为可量化的基准清单。

### 方案一：LLM 生成基准清单（推荐）

**流程**：
1. 在 Phase 1.6（prompt-craft）阶段，让 LLM 同时输出 `baseline.json`
2. `baseline.json` 包含 LLM 认为这本书"应该有"的实体清单：
   ```json
   {
     "characters": {
       "must_have": ["段誉", "萧峰", "虚竹", "王语嫣", "阿朱", "阿紫", ...],
       "should_have": ["段正淳", "刀白凤", "钟灵", "木婉清", ...],
       "nice_to_have": ["褚万里", "古笃诚", ...]
     },
     "factions": {
       "must_have": ["丐帮", "少林寺", "大理段氏", "姑苏慕容氏", "星宿派", ...]
     },
     "skills": {
       "must_have": ["六脉神剑", "北冥神功", "降龙十八掌", "凌波微步", "斗转星移", ...]
     },
     "key_events": {
       "must_have": [
         {"event": "聚贤庄血战", "chapter_range": [17, 20]},
         {"event": "雁门关真相", "chapter_range": [21, 23]},
         ...
       ]
     },
     "key_dialogues": {
       "must_have": [
         {"speaker": "萧峰", "approximate": "我萧峰大好男儿，竟与你这种人齐名"},
         ...
       ]
     }
   }
   ```
3. Phase 3 校验时，自动对比 KB 与 baseline.json，输出覆盖率报告

**优点**：
- 利用 LLM 的先验知识，不依赖人工标注
- 可量化（must_have 覆盖率、should_have 覆盖率）
- 可自动化（脚本对比即可）
- 对每本书定制（不同书的 baseline 不同）

**风险**：
- LLM 可能遗漏某些实体（但 must_have 清单通常不会遗漏经典作品的核心元素）
- LLM 可能列出当前书没有的实体（跨书混淆）→ 需要与 mention_summary 交叉验证

### 方案二：基于 mention_summary 的覆盖率指标（已有基础）

**现状**：`coverage-gap.js` 已实现，但只输出缺口列表，没有量化指标。

**改进**：
1. 定义覆盖率分级：
   - **核心覆盖率**：mention_summary 中 mentions ≥ 100 的术语，KB 覆盖率应 ≥ 95%
   - **重要覆盖率**：mentions ≥ 20 的术语，KB 覆盖率应 ≥ 80%
   - **一般覆盖率**：mentions ≥ 5 的术语，KB 覆盖率应 ≥ 60%
2. 自动计算并输出到质量报告

**优点**：基于原文数据，客观可量化
**缺点**：mention_summary 本身可能不全（compact-mention.js 的提取质量依赖词典）

### 方案三：交叉基准验证（多本书互验）

**流程**：
1. 对同一作者的多本书，提取共同角色/武功/门派的描述
2. 检查一致性（如"降龙十八掌"在天龙八部和射雕英雄传中的描述是否一致）
3. 输出跨书一致性报告

**适用场景**：金庸作品间有大量共享元素（武功、门派、历史背景）
**当前状态**：尚未实现，需要多本书完成后才能做

## 推荐的综合质量指标体系

### 一级指标（必须通过，任一不通过则不发布）

| 指标 | 阈值 | 检查方式 |
|------|------|----------|
| cross-validation errors | = 0 | cross-validate.js |
| locate 率 | ≥ 99% | locate.js |
| 必备实体覆盖率 | ≥ 95% | baseline.json 对比 |
| 跨书混淆 | = 0 | cross-validate.js 幻觉检测 + review-all.md |

### 二级指标（应达到，未达标需人工复核）

| 指标 | 阈值 | 检查方式 |
|------|------|----------|
| grounded 率 | ≥ 80% | verify.js |
| dialogues 真实率 | ≥ 70% | verify_dialogues.js |
| relationship 对称率 | ≥ 95% | cross-validate.js |
| 核心角色对话覆盖率 | ≥ 50% | 统计 dialogues.json 中 must_have 角色的对话占比 |

### 三级指标（参考，用于质量改进）

| 指标 | 说明 | 检查方式 |
|------|------|----------|
| should_have 实体覆盖率 | ≥ 80% | baseline.json 对比 |
| anchor 质量评分 | 平均分 ≥ 7/10 | 方向 J 的 anchor 质量标准 |
| chapter_summaries 事件匹配 | ≥ 90% | Phase 2.2 交叉验证 |

## 实施计划

### Step 1：实现 baseline.json 生成（修改 prompt-craft.md）

在 Phase 1.6 的 meta-prompt 中增加一个输出要求：
- LLM 读取 manifest.json + mention_summary.json + 原文采样
- 除了生成 4 个专属 prompt，还输出 `baseline.json`
- baseline.json 包含 must_have / should_have / nice_to_have 三级清单

### Step 2：实现 baseline-check.js 脚本

```bash
node <skill>/scripts/baseline-check.js <novelDir>
```

功能：
1. 读取 baseline.json
2. 读取 8 个 JSON
3. 对比 must_have 清单与实际 KB，输出覆盖率
4. 检查 baseline 中的实体是否在 mention_summary 中（防止 LLM 列出不存在的实体）
5. 输出 `baseline_report.json`

### Step 3：更新 SKILL.md 质量指标

将一级/二级/三级指标写入 SKILL.md，作为质量门禁。

### Step 4：对已完成的 7 本书回溯验证

用 baseline-check.js 对天龙八部、碧血剑等已完成书籍跑一遍，验证指标体系的合理性。

## 关键文件

- `.agents/skills/generate-kb/prompts/prompt-craft.md` — 需增加 baseline.json 输出要求
- `.agents/skills/generate-kb/SKILL.md` — 需更新质量指标体系
- `.agents/skills/generate-kb/scripts/` — 新增 baseline-check.js
- 各书籍的 `baseline.json` — 新产物

## 验证方式

1. 对天龙八部跑 baseline-check.js，确认 must_have 覆盖率 ≥ 95%
2. 对碧血剑跑 baseline-check.js，验证指标体系对不同书籍的适用性
3. 人工抽样 10 个 must_have 实体，确认 LLM 的 baseline 清单合理性
